# 01. Requirements and Scope

## 코드 주석 원칙

이 문서의 범위에 따라 코드를 작성할 때는 발표자가 핵심 로직을 설명할 수 있도록 한국어 주석을 친절하게 남깁니다. 특히 MVP에서 선택한 제한, 제외 범위, 기존 SQL 엔진을 재사용하는 이유가 코드에 드러나는 위치에 명확한 의도 주석을 둡니다.

이 문서는 과제의 목적, 필수 요구사항, 구현 우선순위, 제외 범위를 고정합니다.

## 과제 목적

기존 메모리 기반 SQL 처리기를 외부에서 호출 가능한 미니 DBMS API 서버로 확장합니다. 결과물은 단순히 동작하는 서버가 아니라, 팀원이 핵심 로직을 설명할 수 있는 구조여야 합니다.

## 필수 요구사항

- C 언어로 API 서버를 구현합니다.
- 외부 클라이언트가 API를 통해 DBMS 기능을 사용할 수 있어야 합니다.
- 요청이 들어오면 스레드 풀의 worker가 SQL 요청을 처리합니다.
- 내부 DB 엔진은 기존 `sql_processor`의 SQL 처리기, `Table`, B+Tree 인덱스를 활용합니다.
- 단위 테스트와 API 기능 테스트를 작성하고 검증 결과를 남깁니다.
- 발표는 별도 슬라이드보다 README 중심으로 진행할 수 있게 준비합니다.

## 구현 우선순위

1. 기존 SQL 처리기 빌드와 테스트를 깨지 않기.
2. 단일 클라이언트가 HTTP API로 SQL을 실행할 수 있게 만들기.
3. SQL 실행 결과를 클라이언트가 읽기 쉬운 응답으로 반환하기.
4. 스레드 풀과 작업 큐를 붙여 병렬 요청을 처리하기.
5. 공유 DB 엔진을 mutex로 보호해 데이터 손상을 막기.
6. API 테스트와 동시성 테스트를 추가하기.
7. README에 실행 방법, API 예시, 테스트 결과, 발표 포인트를 정리하기.

## MVP 범위

- 지원 SQL은 기존 엔진이 지원하는 범위로 제한합니다.
  - `INSERT INTO users VALUES ('Alice', 20);`
  - `SELECT * FROM users;`
  - `SELECT * FROM users WHERE id = 1;`
  - `SELECT * FROM users WHERE id >= 10;`
  - `SELECT * FROM users WHERE name = 'Alice';`
  - `SELECT * FROM users WHERE age > 20;`
- API는 `POST /query` 하나로 시작합니다.
- 저장소는 메모리 기반으로 유지합니다.
- 서버 실행 중 하나의 `users` 테이블을 공유합니다.

## 제외 범위

- SQL 문법 전체 구현은 하지 않습니다.
- 영속 저장소, WAL, transaction, rollback은 구현하지 않습니다.
- 복잡한 REST 리소스 분리는 MVP 이후 확장으로 둡니다.
- 인증, TLS, 사용자별 권한은 구현하지 않습니다.
- 고성능 read/write lock 최적화는 MVP 이후 개선으로 둡니다.

## 설명 가능해야 하는 핵심

- 왜 기존 `sql_execute()`를 API 서버의 DB 진입점으로 삼았는가.
- 왜 스레드 풀을 사용했고, 요청마다 새 스레드를 만들지 않았는가.
- 왜 공유 `Table`과 B+Tree 접근을 mutex로 보호해야 하는가.
- API 요청이 서버 안에서 어떤 순서로 SQL 결과가 되는가.
