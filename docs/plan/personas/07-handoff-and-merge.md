# Handoff And Merge Protocol

## 목적

여러 페르소나가 동시에 일할 때 결과를 안전하게 합치기 위한 규칙을 정리한다.

## 기본 원칙

- 한 파일은 한 페르소나가 주로 책임진다
- 계약이 먼저 정해진다
- 구현은 카드 단위로 나눈다
- handoff note 없이 넘기지 않는다

## handoff note 형식

각 페르소나는 작업 후 아래 내용을 남긴다.

- 변경한 파일
- 바뀐 계약
- 남은 작업
- 주의할 점
- 다음 페르소나가 건드리면 안 되는 부분

## merge 순서

1. Coordinator가 contract file을 먼저 확인한다
2. API와 DB의 경계가 맞는지 본다
3. Concurrency가 API/DB와 충돌하지 않는지 본다
4. QA가 안정성 검증을 한다
5. Docs가 README와 데모를 맞춘다

## 충돌 해결 규칙

- 같은 파일 충돌이면 coordinator가 우선권을 가진다
- 계약 충돌이면 기능보다 계약을 먼저 고친다
- 테스트 실패가 생기면 구현보다 원인을 먼저 찾는다

## 좋은 handoff 예시

- "이 파일은 응답 포맷만 바뀌었고, 에러 코드는 그대로다"
- "큐 full 응답은 503으로 고정했다"
- "DB mutex 범위는 SQL execution 동안만 유지한다"

## 나쁜 handoff 예시

- "대충 됨"
- "다음 사람이 알아서"
- "아직 확정 안 됨"
