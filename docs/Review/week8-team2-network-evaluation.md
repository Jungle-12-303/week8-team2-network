# week8-team2-network 과제 기준 평가 리포트

작성일: 2026-04-22 22:30:42 +09:00

## 평가 기준

이 문서는 [`docs/resources/guideLine.md`](../resources/guideLine.md)의 요구사항을 기준으로 현재 시점의 `week8-team2-network` 프로젝트가 얼마나 잘 구성되어 있는지 평가한 결과를 정리한다.

평가 관점은 다음과 같다.

- 미니 DBMS API 서버 구현 여부
- 기존 SQL 처리기와 B+Tree 재사용 여부
- 스레드 풀 및 동시성 처리 방식
- 단위 테스트, HTTP 테스트, 수동 검증 문서화 수준
- 발표와 README 기준의 전달 가능성
- 과제의 핵심 요구사항 충족도

## 한 줄 총평

현재 프로젝트는 과제 가이드라인의 핵심을 전반적으로 잘 충족하고 있다.  
특히 `POST /query` 기반 API 서버, 기존 `sql_processor` 재사용, 자동화된 테스트 층위 분리, 수동 확인 문서화가 잘 맞물려 있어서, 기능 구현과 검증 체계가 함께 갖춰진 상태로 평가할 수 있다.

| 내부 DB 엔진 재사용 | 매우 우수 | 기존 `sql_processor`와 B+Tree 인덱스를 그대로 활용한다. |

| 운영 편의성 | 보통 이상 | 자동화는 충분하지만, 초기 데이터 리셋과 일부 예시 정리는 더 다듬을 여지가 있다. |

## 요구사항 대비 평가

### 1) API를 통해 DBMS 기능을 사용할 수 있어야 한다

충족한다.

- 서버는 `POST /query`를 중심으로 동작한다.
- SQL 본문을 받아 실행하고 JSON으로 응답한다.
- `README.md`와 테스트 문서에서 사용 흐름이 함께 설명된다.

이 부분은 과제의 핵심 요구사항을 가장 직접적으로 만족한다.

### 2) 스레드 풀을 구성하고 요청마다 스레드를 할당해야 한다

충족한다.

- README에 제한된 작업자 스레드 풀 기반 구조가 명시되어 있다.
- 동시성 문서와 스트레스 테스트가 존재해, 단순 단일 스레드 서버가 아님을 확인할 수 있다.
- 종료 처리와 worker 정리까지 검증 문서가 연결되어 있다.

다만, 발표나 코드 설명에서는 “스레드 풀의 큐 포화 처리”와 “worker 종료 흐름”을 함께 설명하면 설득력이 더 높아진다.

### 3) 기존 SQL 처리기와 B+Tree 인덱스를 그대로 활용해야 한다

충족한다.

- API 서버는 SQL 파서를 새로 만들지 않고 `sql_processor`를 연결해서 쓴다.
- 단위 테스트도 SQL 엔진 레벨에서 직접 검증한다.
- 조건 조회, 정렬, 결과 묶음 같은 내부 엔진 동작이 별도 문서와 테스트로 연결돼 있다.

이 과제에서 “DBMS 자체를 새로 만드는 것”이 아니라 “기존 엔진을 API로 잘 감싸는 것”이라는 방향성을 잘 살린 편이다.

### 4) 단위 테스트와 기능 테스트가 있어야 한다

충족한다.

현재 테스트 층은 다음처럼 나뉜다.

- SQL 단위 테스트
- HTTP 스모크 테스트
- HTTP 통합 테스트
- HTTP 프로토콜 경계값 테스트
- HTTP 타임아웃 테스트
- 수동 쿼리 확인
- 버킷 락 동시성 테스트

좋은 점은 각 테스트가 역할이 다르다는 것이다.

- 단위 테스트는 내부 로직의 정합성을 본다.
- 스모크 테스트는 기본 HTTP 성공 경로를 본다.
- 통합 테스트는 상태 코드와 경계값을 본다.
- 타임아웃 테스트는 장시간 요청이 시스템을 묶지 않는지 본다.
- 수동 쿼리는 사람이 직접 결과를 확인하는 발표/디버깅용 창구 역할을 한다.
- 동시성 테스트는 현재 구현된 버킷 락이 실제로 병렬성을 확보하는지 본다.

### 5) 발표와 README 기준 설명이 가능해야 한다

충족한다.

- 루트 README가 전체 프로젝트를 한 번에 설명한다.
- `docs/test/demo/README.md`는 테스트 기록표와 실행 순서를 제공한다.
- `docs/Review`는 평가와 회고를 기록하는 공간으로 나뉘어 있다.
- `docs/study`는 해결 기록을 남기는 공간으로 분리되어 있다.

즉, 발표할 때 “무엇을 만들었는지”와 “어떻게 검증했는지”를 따로 보여주기 쉽다.

## 잘 구성된 점

### 1) 프로젝트 구조가 역할별로 분리되어 있다

현재 레포는 다음처럼 역할이 분명하다.

- `server/`는 HTTP 진입점과 요청 처리
- `sql_processor/`는 DB 내부 엔진
- `scripts/`는 실행 가능한 테스트
- `docs/plan/`은 설계와 계획
- `docs/test/demo/`는 실제 테스트 안내
- `docs/Review/`는 평가와 회고
- `docs/study/`는 문제 해결 기록

이 분리는 과제 발표뿐 아니라, 나중에 리뷰를 받을 때도 구조를 설명하기 좋다.

### 2) 테스트가 “한 번 돌려보는 수준”을 넘는다

테스트는 자동화와 수동 확인이 같이 있다.

- 자동화는 재현성과 신뢰도를 높인다.
- 수동 확인은 출력 가독성과 사람이 보는 UX를 보강한다.
- 동시성/타임아웃/프로토콜 경계값까지 들어가 있어서 단순 기능 확인보다 훨씬 탄탄하다.

특히 `docs/test/demo/README.md`의 테스트 기록표는

- 실행 스크립트
- 테스트 안내
- 규격 문서

를 함께 연결해 두어, “어떤 테스트를 왜 돌리는지”를 추적하기 쉽다.

### 3) 현재 구현 기준에 맞춘 문서 갱신이 꾸준하다

예전에는 `rwlock`나 전역 락처럼 옛 용어가 섞인 부분이 있었지만, 지금은 그 부분을 버킷 락 기준으로 다시 정리한 상태다.

즉, 문서가 단순 설명서가 아니라 실제 구현 변화에 맞춰 계속 조정되고 있다.

### 4) 수동 확인 경로가 분명하다

수동 쿼리 문서가 따로 있고, 스모크 테스트 출력도 사람이 읽기 좋게 정리되어 있다.

이건 발표 때 특히 좋다.

- “자동 테스트는 여기서 돌렸다.”
- “사람 눈으로 보는 확인은 여기서 했다.”

이 두 축을 분리해서 보여줄 수 있기 때문이다.

## 아쉬운 점

### 1) 메모리 기반 데이터가 누적되기 쉽다

서버가 메모리 기반이라, 같은 프로세스를 오래 돌리면 row가 계속 누적될 수 있다.

이건 기능 버그라기보다 사용성 문제에 가깝다.

개선 아이디어는 다음과 같다.

- 테스트 전용 reset 명령
- 샘플 데이터 초기화 모드
- 테스트용 서버 재시작 절차 자동화

### 2) 일부 문서 파일명은 아직 예전 용어를 유지한다

예를 들어 `rwlock-test.md`는 현재 내용상 버킷 락 테스트 문서에 가깝다.  
내용은 맞지만, 파일명까지 현재 구조에 맞춰 바꾸면 더 직관적이다.

즉, 기능은 맞지만 용어와 이름을 한 번 더 다듬으면 더 좋다.

### 3) 타임아웃 테스트와 동시성 테스트는 발표 때 한 번 더 설명이 필요하다

자동화는 잘 되어 있지만, 발표 자리에서는 다음을 추가로 설명해야 한다.

- 타임아웃이 왜 필요한지
- 버킷 락이 전역 락보다 왜 나은지
- `SELECT`와 `INSERT`가 어떤 방식으로 동시에 검증되는지

이건 구현 부족이 아니라, 발표 전달력 강화를 위한 보완 포인트다.

## 문서/테스트 구성 평가

현재 프로젝트는 문서와 스크립트의 연결성이 좋은 편이다.

- `scripts/tests/`는 실제 실행 단위다.
- `docs/test/demo/tests/`는 각 실행 단위에 대한 설명이다.
- `docs/test/demo/README.md`는 기록표와 실행 순서를 제공한다.

이 구조 덕분에 “무엇을 실행해야 하는지”와 “실행 결과를 어디에 적는지”가 분리되어 있다.

다만, 다음 개선은 고려할 만하다.

- 파일명 정리
- 예시 출력 일부 간소화
- 초기 데이터 리셋 도구 추가

## 최종 판단

현재 `week8-team2-network` 프로젝트는 `docs/resources/guideLine.md` 기준으로 볼 때 **핵심 요구사항을 잘 충족한 프로젝트**다.

특히 강점은 다음과 같다.

- API 서버로의 구조 전환이 분명하다
- 기존 SQL 엔진과 B+Tree를 잘 재사용한다
- 테스트가 자동화와 수동 확인으로 나뉘어 있다
- 동시성, 타임아웃, 프로토콜 경계값까지 검증하고 있다
- 문서가 발표와 데모에 바로 쓰기 좋다

남은 개선 포인트는 주로 완성도와 전달력이다.

- 데이터 초기화 편의성
- 파일명과 용어의 일관성
- 발표용 예시 압축

결론적으로, 이 프로젝트는 “과제를 제출할 수준”을 넘어 “설명과 검증까지 함께 보여줄 수 있는 수준”으로 정리되어 있다고 평가할 수 있다.
