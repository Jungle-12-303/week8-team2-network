# week8-team2-network 과제 기준 평가 리포트

작성일: 2026-04-22 17:09:56 +09:00

## 범위

이 문서는 `docs/resources/guideLine.md`의 요구사항을 기준으로 `week8-team2-network` 프로젝트를 평가한 결과를 정리한다.

평가 대상은 다음과 같다.

- 미니 DBMS API 서버 구현 여부
- 기존 `sql_processor`와 B+Tree 재사용 정도
- 스레드 풀 및 요청 처리 구조
- 단위 테스트 / 기능 테스트 / 수동 검증 문서화
- 발표와 README 기준의 전달 가능성

## 한 줄 총평

과제의 핵심 요구사항을 대부분 충실하게 충족했고, 특히 HTTP API와 SQL 처리기의 연결, 테스트 자동화, 수동 확인 문서가 잘 갖춰져 있다.  
동시성 측면에서는 읽기/쓰기 락 분리까지 들어가 있어 기본기는 좋고, 데이터 초기화와 부하 검증을 더 명확히 하면 완성도가 한 단계 더 올라간다.

| 내부 DB 엔진 재사용 | 매우 우수 | 기존 `sql_processor`와 B+Tree를 서버 계층에서 그대로 활용한다. |

## 잘한 점

### 1) 과제 요구사항을 정확히 찌른다

이 프로젝트는 외부 클라이언트가 `POST /query`로 SQL을 보내고, 서버가 이를 `sql_processor`로 넘겨 실행한 뒤 JSON으로 돌려주는 흐름을 명확하게 구현한다.

즉, 과제에서 요구한 핵심은 다음과 같이 충족된다.

- C 기반 API 서버
- SQL 요청을 HTTP로 받는 단일 엔드포인트
- 기존 SQL 처리기 재사용
- B+Tree 인덱스 활용
- Docker 기반 실행 지원

### 2) 테스트 자동화가 단단하다

테스트가 단순히 “한 번 되는지 보는 수준”이 아니라 층위가 나뉘어 있다.

- `sql_processor/unit_test.c`
- `scripts/smoke_test.sh`
- `scripts/http_integration_test.sh`
- `scripts/manual_query.sh`

특히 좋은 점은 다음과 같다.

- 단위 테스트는 SQL 엔진 자체를 검증한다.
- 스모크 테스트는 HTTP 경로에서 `INSERT` / `SELECT`의 최소 동작을 확인한다.
- 통합 테스트는 `405`, `404`, `413` 같은 경계값까지 본다.
- 수동 쿼리 도구는 서버를 직접 띄우고 사람이 입력해 확인하는 흐름을 지원한다.

### 3) 문서 구조가 실제 사용 시나리오에 맞는다

문서가 계획용, 설명용, 테스트용으로 분리되어 있어서 읽는 사람이 흐름을 따라가기 쉽다.

- `docs/plan/`은 기획과 설계
- `docs/test/demo/`는 실제 실행 및 수동 확인
- `README.md`는 진입점

이 구조 덕분에 발표 때도 “무엇을 만들었는지”와 “어떻게 검증했는지”를 분리해서 설명하기 좋다.

### 4) 수동 검증 경험이 좋다

`scripts/manual_query.sh` 덕분에 서버를 띄운 뒤 직접 `INSERT`, `SELECT`를 쳐 보면서 응답을 볼 수 있다.  
이건 단순한 자동화보다도 발표와 디버깅에 유리하다.

## 아쉬운 점

### 1) 동시성은 안전하면서도 읽기/쓰기 분리가 되어 있다

처음에는 공유 DB를 단일 mutex로만 막는다고 봤지만, 실제 코드를 다시 확인해 보니 `server/api.c`에서 `pthread_rwlock_t`를 사용하고 `sql_determine_lock_mode()`에 따라 읽기 락과 쓰기 락을 분리하고 있다.

즉, 이 프로젝트의 동시성 평가는 다음처럼 수정하는 것이 정확하다.

- `SELECT`는 읽기 락 경로를 탄다.
- `INSERT`는 쓰기 락 경로를 탄다.
- 서버 전체가 단일 mutex로만 직렬화되지는 않는다.

그래서 이전에 적었던 “읽기 요청까지 같이 막힌다”는 평가는 이 코드에는 맞지 않는다.  
대신 여전히 보완 여지는 있다.

- 락 경로가 SQL 첫 키워드 판단에 의존하므로, 더 정교한 문장 분석이 있으면 안전성이 좋아질 수 있다.
- 큐 포화 상황에 대한 더 명시적인 테스트는 추가할 가치가 있다.
- `SELECT *`와 조건 조회의 병렬성을 실제 부하 상황에서 더 보여주면 설득력이 높아진다.

### 2) 메모리 기반 상태가 누적되기 쉽다

서버가 메모리 기반이라 실행 중인 프로세스 안에서는 row가 계속 누적된다.  
그래서 수동 테스트를 오래 하다 보면 결과가 길어지고, 출력이 읽기 어려워질 수 있다.

현재는 문서에서 서버 재시작을 안내하고 있지만, 더 편하게 하려면 다음이 있으면 좋다.

- 테스트 전용 reset 명령
- 샘플 데이터 초기화 모드
- 스모크 테스트가 자체적으로 깨끗한 상태를 만드는 방식

### 3) 수동 테스트 예시는 더 줄일 여지가 있다

문서는 이미 충분히 친절하지만, 발표 직전에는 예시가 너무 많아도 오히려 핵심이 흐려질 수 있다.  
지금은 다양한 쿼리를 보여주는 데 강점이 있지만, 발표용으로는 대표 케이스 2~3개만 남겨도 충분하다.

## 요구사항 충족도 평가

### 1. 외부 클라이언트가 API를 통해 DBMS 기능을 사용할 수 있어야 한다

충족했다.  
`POST /query`로 SQL을 전달하고 JSON 응답을 받는 구조가 구현되어 있다.

### 2. 스레드 풀을 구성하고 요청마다 스레드를 할당해야 한다

충족했다.  
구현 자체는 스레드 풀 기반이며, 요청 처리 흐름도 문서와 코드에서 확인할 수 있다.

### 3. 기존 SQL 처리기와 B+Tree 인덱스를 활용해야 한다

충족했다.  
API 서버는 SQL 엔진을 새로 만들지 않고 기존 `sql_processor`를 감싸는 방식이다.

### 4. 단위 테스트와 API 기능 테스트가 있어야 한다

충족했다.  
단위 테스트, 스모크 테스트, 통합 테스트가 모두 존재하고, 실행 방법도 문서화되어 있다.

### 5. 발표와 README 중심 설명이 가능해야 한다

충족했다.  
README가 구조화되어 있고, 테스트와 수동 확인 문서도 별도 정리되어 있어 발표 자료로 쓰기 좋다.

## 최종 판단

이 프로젝트는 과제 가이드라인을 기준으로 보면 “핵심 요구사항을 안정적으로 충족한 팀”으로 평가할 수 있다.

특히 다음 점이 강하다.

- API 서버로의 전환이 분명하다
- 기존 SQL 엔진 재사용이 자연스럽다
- 테스트 자동화와 수동 검증이 같이 준비되어 있다
- 문서가 실행 흐름과 발표 흐름을 함께 지원한다

남은 개선 포인트는 주로 성능과 운영 편의성이다.

- 읽기 병렬성 개선
- 데이터 초기화 편의성
- 부하/동시성 스트레스 테스트 확장

즉, 기능 완성도는 높고, 이제는 “더 잘 보이게 하고 더 잘 설명하기” 단계에 들어간 프로젝트라고 볼 수 있다.
