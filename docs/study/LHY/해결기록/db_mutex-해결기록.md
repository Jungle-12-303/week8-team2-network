# SQL 실행 병렬성 문제 정리

## 1. 문제 정의

현재 서버는 worker thread pool로 요청을 분산하지만, 실제 SQL 실행은 `db_mutex` 하나로 보호되고 있다.

즉, 여러 worker thread가 동시에 일을 받아도 DB 접근 자체는 한 번에 하나씩만 처리되는 구조다.

```text
client request
  -> worker thread
  -> api_handle_query()
  -> pthread_mutex_lock(db_mutex)
  -> sql_execute()
  -> pthread_mutex_unlock(db_mutex)
```

이 구조에서는 요청 처리 병렬성은 있어도 SQL 실행 병렬성은 거의 없다.

### mutex는 무엇인가

`mutex`는 `mutual exclusion`의 줄임말로, 보통 `상호 배제 락`이라고 부른다.

뜻은 아주 간단하다.

- 같은 자원을 여러 스레드가 동시에 건드리지 못하게 막는다
- 한 스레드가 쓰는 동안에는 다른 스레드가 기다린다
- 사용이 끝나면 다음 스레드가 들어간다

쉽게 말하면 `mutex`는 화장실 열쇠처럼 생각하면 된다.

- 열쇠를 가진 사람만 들어갈 수 있다
- 누군가 쓰는 중이면 다른 사람은 밖에서 기다린다
- 다 쓰고 나오면 다음 사람이 들어간다

DB에서 `mutex`를 쓰는 이유는 데이터가 꼬이지 않게 하려는 것이다.
예를 들어 여러 스레드가 동시에 같은 테이블에 `INSERT`를 하면, 중간 상태가 섞이거나 잘못된 결과가 나올 수 있다.

### rwlock은 어떻게 다른가

`rwlock`은 `read-write lock`의 줄임말이다.

이름 그대로 읽기와 쓰기를 나눠서 제어한다.

- 읽기 락은 여러 스레드가 동시에 잡을 수 있다
- 쓰기 락은 오직 하나만 잡을 수 있다
- 쓰기 락이 잡혀 있으면 읽기 락도 못 잡는다
- 읽기 락이 하나라도 잡혀 있으면 쓰기 락은 기다려야 한다

즉, 질문으로 바꾸면 이렇게 된다.

- 읽을 때는 읽기 락을 여러 스레드가 같이 잡을 수 있다
- 쓸 때는 쓰기 락이 독점한다
- 읽기와 쓰기는 동시에 못 한다

화장실 비유로 보면:

- `mutex`는 방 전체를 한 사람만 쓰는 구조
- `rwlock`은 읽는 사람들은 같이 들어가도 되지만, 쓰는 사람은 혼자 들어가야 하는 구조

그래서 읽기 요청이 많은 DB에서는 `rwlock`이 유리할 수 있다.
읽기끼리는 서로 방해하지 않기 때문이다.

하지만 쓰기 요청이 들어오면, 읽기 요청도 잠깐 멈춰야 한다.
DB 내용이 바뀌는 순간을 안전하게 지켜야 하기 때문이다.

### 이 프로젝트에서 왜 중요하나

지금 구조는 `SELECT`도 `INSERT`도 같은 `db_mutex`를 사용한다.  
그래서 읽기 요청끼리도 동시에 처리하지 못하고, 모두 순서대로 대기한다.

이걸 `rwlock`으로 바꾸면:

- `SELECT`끼리는 같이 처리 가능
- `INSERT`는 독점적으로 처리
- 읽기 요청이 많은 상황에서 전체 처리량이 올라갈 수 있음

즉, `rwlock`은 "쓰는 동안은 읽지 않고, 읽는 동안은 쓰지 않는" 규칙을 더 정교하게 구현한 락이라고 보면 된다.

### 왜 병목이 되는가

`db_mutex`는 단순히 테이블 접근만 막는 것이 아니라, `sql_execute()` 전체를 감싸고 있다.

즉 아래 작업도 모두 잠긴 상태에서 진행된다.

- SQL 파싱
- 조건 해석
- 결과 생성
- 테이블 접근

이렇게 되면 worker thread pool의 장점이 크게 줄어든다.

### 병렬로 처리할 수 있는 요청도 순서대로 막힌다

예를 들어 다음 두 요청은 서로 다른 데이터를 보는 경우가 많다.

- `SELECT * FROM users WHERE id = 1`
- `SELECT * FROM users WHERE id = 999`

이론적으로는 동시에 처리해도 문제없는 경우가 많지만, 지금 구조에서는 둘 다 같은 mutex를 기다려야 한다.

## 2. 왜 병목이 되는가

### 2-1. 읽기와 쓰기가 같은 락을 공유한다

현재는 `SELECT`와 `INSERT` 모두 같은 mutex를 사용한다.

그래서 다음 요청들이 전부 같은 줄에 선다.

- `INSERT INTO users VALUES (...)`
- `SELECT * FROM users`
- `SELECT * FROM users WHERE id = ...`
- `SELECT * FROM users WHERE age > ...`

읽기끼리는 충돌하지 않는 경우가 많은데도, 모두 차례를 기다린다.

### 2-2. 락 범위가 너무 넓다

`db_mutex`는 단순히 테이블 접근만 막는 것이 아니라, `sql_execute()` 전체를 감싸고 있다.

즉 아래 작업도 모두 잠긴 상태에서 진행된다.

- SQL 파싱
- 조건 해석
- 결과 생성
- 테이블 접근

이렇게 되면 worker thread pool의 장점이 크게 줄어든다.

### 2-3. 병렬로 처리할 수 있는 요청도 순서대로 막힌다

예를 들어 다음 두 요청은 서로 다른 데이터를 보는 경우가 많다.

- `SELECT * FROM users WHERE id = 1`
- `SELECT * FROM users WHERE id = 999`

이론적으로는 동시에 처리해도 문제없는 경우가 많지만, 지금 구조에서는 둘 다 같은 mutex를 기다려야 한다.

## 3. mutex를 꼭 써야 하는 이유와 한계

mutex를 쓰는 이유는 분명하다.

- 여러 스레드가 동시에 같은 메모리를 망가뜨리지 않게 하려는 것
- 테이블, 배열, B+Tree 같은 공유 자원을 안전하게 보호하려는 것

즉, mutex 자체가 나쁜 것은 아니다.  
문제는 `어디까지 잠그느냐`와 `얼마나 오래 잠그느냐`다.

지금은 다음처럼 되어 있다.

- 필요한 부분만 잠그지 않는다
- 읽기 요청도 쓰기 요청처럼 취급한다
- SQL 파싱까지 잠금 범위에 들어간다

그래서 안전은 확보되지만, 성능이 크게 희생된다.

---

## 4. 코드에서 확인한 근거

- `server/api.c`에서 `api_handle_query()`가 `db_mutex`를 lock한 뒤 `sql_execute()`를 호출한다.
- `server/server.c`는 요청을 worker pool에 넘기지만, SQL 처리 구간 자체는 따로 병렬화하지 않는다.
- `sql_processor/sql.c`는 `INSERT`와 `SELECT`를 같은 실행 경로로 처리한다.
- `sql_processor/table.c`는 `id` 조건에서 B+Tree를 활용하지만, 상위 mutex가 있으면 그 장점이 줄어든다.

## 5. 해결 방법 후보

### 5-1. `pthread_rwlock_t`로 교체

가장 현실적인 첫 번째 개선안이다.

#### 방식

- `SELECT`는 read lock
- `INSERT`는 write lock

#### 장점

- 구현 난이도가 비교적 낮다
- 읽기 요청이 많을 때 동시성이 좋아진다
- 현재 코드 구조를 크게 바꾸지 않아도 된다

#### 단점

- `SELECT`가 진짜 읽기 전용이어야 한다
- 여전히 테이블 단위 락이라 세밀한 병렬성은 제한된다

---

### 5-2. 파싱과 실행 락 분리

SQL 문자열 해석은 lock 밖에서 하고, 실제 table 접근만 lock을 건다.

#### 장점

- 락 점유 시간이 줄어든다
- 대기열이 짧아진다

#### 단점

- 코드 구조를 조금 더 잘 나눠야 한다
- 에러 경로마다 unlock이 반드시 보장되어야 한다

---

### 5-3. 테이블 / row / index 단위로 세분화

더 큰 개선이지만 설계 난이도도 높다.

#### 장점

- 병렬성 확장 여지가 크다

#### 단점

- 락 순서와 범위를 잘못 잡으면 데드락이 생길 수 있다
- 현재 프로젝트 범위에서는 과할 수 있다

---

### 5-4. MVCC 또는 snapshot 방식

읽기 요청이 거의 막히지 않는 구조다.

#### 장점

- 읽기/쓰기 동시성이 매우 좋아진다

#### 단점

- 구현 난이도가 높다
- 메모리 관리가 복잡해진다
- 지금 프로젝트 범위에는 과한 편이다

---

## 6. 추천 방향

이 프로젝트에서는 아래 순서가 가장 현실적이다.

1. `db_mutex`를 `pthread_rwlock_t`로 바꾸기
2. `SELECT`는 read lock, `INSERT`는 write lock 사용
3. SQL 파싱은 lock 밖으로 빼기
4. 이후 필요하면 테이블/인덱스 단위로 더 세분화하기

---

## 7. 구현 시 주의점

- `SELECT`가 정말 읽기 전용인지 먼저 확인해야 한다.
- 여러 락을 쓸 경우 락 순서를 고정해야 한다.
- 에러가 나도 unlock이 반드시 실행되어야 한다.
- 바꾼 뒤에는 단일 요청뿐 아니라 동시 요청 throughput도 실제로 측정해야 한다.

---

## 8. 결론

현재 병목의 본질은 worker thread가 없어서가 아니라,  
**worker thread가 있어도 DB 접근이 하나의 mutex로 직렬화된다**는 점이다.

가장 먼저 적용할 개선책은 `pthread_rwlock_t` 도입이다.  
이 방식이면 현재 구조를 크게 흔들지 않으면서도 읽기 동시성을 바로 얻을 수 있다.
