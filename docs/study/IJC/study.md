# 미니 DBMS API 서버 - 학습 가이드

> 이 가이드는 코드 디테일보다는 **전체 구조, 흐름, 핵심 개념**을 이해하는 데 초점을 맞춥니다.

---

## 📌 프로젝트의 목표

**"외부 클라이언트가 HTTP API를 통해 데이터베이스에 접근할 수 있는 서버를 만든다"**

```
사용자의 요청
    ↓
HTTP API 서버 ← 외부 클라이언트는 여기로 요청
    ↓
SQL 처리 엔진 ← 이전 프로젝트에서 만든 것 재사용
    ↓
메모리 기반 데이터베이스
    ↓
결과를 JSON으로 응답
```

## 🔄 요청-응답 흐름

### Step 1: 클라이언트가 HTTP 요청을 보낸다

```bash
curl -X POST http://localhost:8080/query \
  -H "Content-Type: text/plain" \
  --data "INSERT INTO users VALUES ('Alice', 20);"
```

**무엇이 일어나는가:**
- 클라이언트가 `POST /query` 로 요청을 전송
- HTTP 헤더 + 바디(SQL 쿼리)를 함께 전송

---

### Step 2: 서버가 요청을 받는다

```
TCP 소켓 바인드 (포트 8080)
    ↓
accept() - 클라이언트 연결 수락
    ↓
클라이언트 fd를 스레드 풀의 작업 큐에 추가
    ↓
워커 스레드가 깨어남
```

**핵심 개념: 스레드 풀**
- **왜 필요한가?** 모든 요청마다 새로운 스레드를 만들면 비효율적
- **어떻게 작동하는가?**
  - 미리 4개의 워커 스레드를 생성
  - 16개 크기의 작업 큐를 준비
  - 요청이 들어오면 큐에 추가
  - 유휴 워커가 큐에서 작업을 꺼내 처리
  - 처리 후 다시 대기

```
[워커1] [워커2] [워커3] [워커4]  ← 항상 대기 중
   ↑      ↑      ↑      ↑
   └──────┼──────┼──────┘
      ↓  ↓  ↓  ↓
    ┌─────────────────┐
    │   작업 큐       │  ← 요청들이 여기 대기
    │ [요청1] [요청2] │
    └─────────────────┘
```

---

### Step 3: HTTP 요청을 파싱한다

```
Raw HTTP 데이터:
POST /query HTTP/1.1
Content-Type: text/plain
Content-Length: 42

INSERT INTO users VALUES ('Alice', 20);

    ↓ (파싱)

HttpRequest {
  method: "POST",
  path: "/query",
  version: "HTTP/1.1",
  content_length: 42,
  body: "INSERT INTO users VALUES ('Alice', 20);"
}
```

**파싱의 핵심:**
- HTTP 헤더에서 `Content-Length`를 읽음
- 정확히 그 바이트 수만큼만 바디를 읽음
- SQL 쿼리 추출

---

### Step 4: SQL을 처리한다

```
SQL 쿼리: "INSERT INTO users VALUES ('Alice', 20);"
    ↓
sql_processor (이전 프로젝트)
    ├─ 쿼리 타입 파악 (INSERT vs SELECT)
    ├─ 문법 검증
    ├─ 테이블에서 작업 수행
    │  ├─ INSERT: 새 레코드 추가
    │  ├─ SELECT: 조건에 맞는 레코드 찾기
    │  └─ 인덱스 활용 (B+ 트리)
    └─ SQLResult 반환
```

**중요한 동시성 보호:**
```
여러 워커가 동시에 DB에 접근할 수 있으므로...

워커1이 DB 접근 시작
  ↓
pthread_mutex_lock(&db_mutex)  ← 다른 워커 차단
  ↓
SQL 실행
  ↓
pthread_mutex_unlock(&db_mutex)  ← 다른 워커 진입 허용

워커2가 이제 접근 가능
```

---

### Step 5: 결과를 JSON으로 변환한다

```
SQLResult (C 구조체):
{
  action: SQL_ACTION_INSERT,
  inserted_id: 1,
  row_count: 1,
  ...
}
    ↓ (직렬화)

JSON:
{
  "ok": true,
  "action": "insert",
  "inserted_id": 1,
  "row_count": 1
}
```

**JSON 버퍼 개념:**
- 동적 문자열을 효율적으로 만들기 위해 버퍼 사용
- 메모리 할당과 해제를 최소화
- 문자열을 조금씩 붙여나감

---

### Step 6: HTTP 응답을 보낸다

```
HTTP 응답:
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 59

{"ok":true,"action":"insert","inserted_id":1,"row_count":1}

    ↓ (TCP로 전송)

클라이언트가 응답 수신
```

## 🎯 핵심 개념 정리

### 1. 동시성 (Concurrency)

**문제:** 여러 요청이 **동시에** 들어올 때 무엇이 어려운가?

```
요청1: INSERT INTO users VALUES ('Alice', 20);
요청2: INSERT INTO users VALUES ('Bob', 25);
요청3: SELECT * FROM users;

    ↓

워커1이 요청1 처리 중
워커2가 요청2 처리 중 (동시에!)
워커3이 요청3 처리 중 (동시에!)

    ↓ 문제 발생!

같은 테이블에 동시에 접근하면?
→ 데이터 손상 가능
→ 읽기/쓰기 순서가 뒤바뀔 수 있음
→ 레코드가 중복되거나 손실될 수 있음
```

**해결책: 뮤텍스 (상호 배제)**

```
╔════════════════════════════════════════╗
║  DB 뮤텍스 (상호 배제 잠금)           ║
╚════════════════════════════════════════╝

워커1: [lock]
       ↓
       DB 접근 (다른 워커 차단됨)
       ↓
       [unlock]

워커2: 대기... 대기...
       ↓
       [lock] (이제 접근 가능)
       ↓
       DB 접근
       ↓
       [unlock]
```

#### 🔐 뮤텍스(Mutex) 개념

**정의:** 여러 스레드가 공유 자원에 접근할 때 **한 번에 하나만** 진입하도록 강제하는 잠금 장치

**어떻게 작동:**
```
요청1이 lock 획득 → DB 작업 → unlock 해제
                                    ↓
요청2는 대기 중... (lock 획득 불가) → unlock 후 lock 획득 → DB 작업
```

**임계 영역 (Critical Section):**
```c
pthread_mutex_lock(&db_mutex);    // 잠금
// ← 이 부분만 보호됨 (다른 스레드 차단)
sql_execute(table, sql);
pthread_mutex_unlock(&db_mutex);  // 해제
```

**주의:** 임계 영역은 **최소한으로** 유지 (길수록 병렬성 저하)

---

### 2. 스레드 풀의 동시성

**스레드 풀 내부에서도 동시성 관리 필요:**

```
작업 큐에 대한 접근도 보호해야 함

┌──────────────────────────────┐
│  작업 큐 (동시 접근 방지)    │
├──────────────────────────────┤
│  [뮤텍스]  큐의 내용 보호     │
│  [조건변수] 대기/깨우기       │
└──────────────────────────────┘

메인 스레드가 요청 추가:
  pthread_mutex_lock(&pool->mutex)
  큐에 요청 추가
  pthread_cond_signal()  ← 워커를 깨움
  pthread_mutex_unlock(&pool->mutex)

워커 스레드가 대기:
  pthread_mutex_lock(&pool->mutex)
  while (큐가 비어있음) {
    pthread_cond_wait()  ← 깨어날 때까지 대기
  }
  큐에서 작업 꺼냄
  pthread_mutex_unlock(&pool->mutex)
```

### 3. 모듈화와 책임 분리

**각 모듈은 하나의 책임만:**

| 모듈 | 책임 |
|------|------|
| `http.c` | HTTP 요청/응답 처리 (네트워크) |
| `api.c` | SQL 실행 + 결과 변환 (비즈니스 로직) |
| `json_util.c` | JSON 직렬화 (포맷 변환) |
| `thread_pool.c` | 워커 관리 (병렬 처리) |
| `server.c` | 전체 조율 (메인 루프) |
| `sql_processor/` | SQL 처리 (이전 프로젝트) |

**장점:**
- 각 모듈을 독립적으로 테스트 가능
- 수정이 필요하면 해당 모듈만 수정
- 코드 재사용 가능

## 🔍 실전 이해 체크

다음 질문에 답할 수 있다면 제대로 이해한 것입니다:

### 동시성
1. **Q:** 두 개의 INSERT 요청이 동시에 들어올 때, 데이터 손상을 방지하려면?
   - **A:** db_mutex를 사용해서 한 번에 하나의 워커만 DB에 접근하도록 함

2. **Q:** 스레드 풀의 워커들이 새 작업을 기다릴 때 뭐를 사용하나?
   - **A:** 조건 변수(pthread_cond_wait)를 사용해서 효율적으로 대기

### 흐름
3. **Q:** 클라이언트의 요청이 어떤 순서로 처리되나?
   - **A:** accept → 큐 추가 → 워커가 처리 → HTTP 응답 → 클라이언트 수신

4. **Q:** HTTP 바디의 크기를 어떻게 알아?
   - **A:** Content-Length 헤더를 읽어서 정확히 그만큼만 읽음

### 아키텍처
5. **Q:** 왜 sql_processor를 재사용하는가?
   - **A:** SQL 처리 로직이 복잡하므로 이미 검증된 코드를 재사용 (DRY 원칙)

6. **Q:** json_util.c는 어떤 역할?
   - **A:** C 구조체를 JSON 문자열로 변환하는 책임을 분리해서 관리
