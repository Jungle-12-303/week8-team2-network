# 미니 DBMS API 서버 - 쉬운 설명

> 어려운 단어 없이 **핵심 아이디어만** 설명합니다.

## 🎯 이 프로젝트가 하는 일

**간단히 말하면:**
- 인터넷을 통해 데이터베이스를 사용할 수 있게 만드는 것
- 마치 웹사이트처럼 `http://...` 주소로 데이터를 저장하고 조회

**비유:**
```
배달앱처럼 생각하면 된다!

손님: "라면 주문할래"
배달앱: (주문 받음)
식당: (라면 준비)
배달앱: (배달)
손님: (라면 받음)

이 프로젝트도:
클라이언트: "데이터 저장해"
API 서버: (요청 받음)
데이터베이스: (저장)
API 서버: (결과 응답)
클라이언트: (결과 받음)
```

## 🏗️ 전체 구조

### 어떤 부분들로 이루어져 있나?

```
┌─────────────────────────────────┐
│    컴퓨터1 (클라이언트)          │
│  curl로 데이터를 달라고 요청     │
└──────────────┬──────────────────┘
               │ 인터넷 연결
               ↓
┌─────────────────────────────────┐
│    컴퓨터2 (이 프로젝트)         │
│  요청을 받아서 처리하는 서버    │
│                                 │
│  ├─ HTTP 담당 (편지 받기)      │
│  ├─ 워커 관리 (여러명 일꾼)    │
│  ├─ SQL 처리 (DB 명령 실행)    │
│  └─ JSON 응답 (답장하기)       │
│                                 │
│     데이터베이스 (데이터 저장)  │
└─────────────────────────────────┘
```

## 🔄 요청이 처리되는 과정

### 단계별로 무엇이 일어나나?

**Step 1: 클라이언트가 요청 전송**
```
curl -X POST http://localhost:8080/query \
  --data "INSERT INTO users VALUES ('Alice', 20);"

= "Alice라는 사람, 나이 20살을 저장해줘"
```

**Step 2: 서버가 요청 받음**
```
손님이 음식점에 들어옴
→ 문을 (accept) 열어서 들여보냄
→ 줄을 서서 대기 (작업 큐)
```

**Step 3: 일꾼이 일처리**
```
대기 중이던 일꾼이 깨어남
→ 손님의 요청을 읽음 (HTTP 파싱)
→ "아, Alice를 저장해야겠네" (SQL 추출)
→ 데이터베이스에 저장 (SQL 실행)
```

**Step 4: 결과를 JSON으로 만듦**
```
저장 완료!
↓
결과를 편지 형태로 쓴다
↓
{"ok": true, "inserted_id": 1}
```

**Step 5: 응답 전송**
```
편지를 클라이언트에게 보냄
↓
클라이언트가 "저장됐어!" 확인
```

## 💡 핵심 개념 3가지

### 1️⃣ 여러 명의 일꾼 (스레드 풀)

**왜 필요한가?**
```
식당에 손님이 100명 들어옴

방법1: 손님마다 새 요리사 채용
→ 비용 낭비, 혼란 (별로!)

방법2: 요리사 4명을 처음부터 채용
→ 손님들이 줄을 서서 기다림
→ 4명이 계속 일함
→ 효율적! (좋아!)
```

**이 프로젝트:**
- 일꾼 4명 준비
- 대기 공간 16명 (줄 세우는 곳)
- 일꾼이 비면 줄 선 사람이 들어옴

### 2️⃣ 동시에 같은 것을 건드리면 망한다 (뮤텍스)

**문제:**
```
Alice 저장 중...
   ↓
Bob도 동시에 저장 시작...
   ↓
데이터가 섞여서 망함!
```

**해결책: 뮤텍스 (빨간 불/초록 불)**
```
┌─────────────┐
│ 데이터베이스 │  (초록 불 = 누구나 진입 가능)
└─────┬───────┘
      │ Alice가 접근 시작
      ↓
┌─────────────┐
│ 데이터베이스 │  (빨간 불 = Alice만 사용 중)
└─────┬───────┘
      │ Bob은 대기...
      │ Alice가 저장 완료
      ↓
┌─────────────┐
│ 데이터베이스 │  (초록 불 = Bob이 접근 가능)
└─────────────┘
```

**쉽게 말하면:**
- 한 번에 한 명만 데이터에 손댈 수 있다
- 나머지는 기다린다
- 끝나면 다음 사람 차례

### 3️⃣ 모듈화 (각자 맡은 일만 한다)

**식당 같은 분업:**
```
웨이터: 손님 말 듣기
요리사: 요리만 하기
캐셔: 돈 받기
청소부: 정리하기

각자 자기 일만 하면 → 효율적!
```

**이 프로젝트도:**
```
http.c      : 네트워크 통신 (편지 받기/보내기)
api.c       : 업무 처리 (뭘 해야 할지 결정)
json_util.c : 형식 변환 (JSON 만들기)
thread_pool : 일꾼 관리
sql_processor : 데이터베이스 실제 처리
```

## 📊 요청이 흐르는 길

```
┌──────────────────┐
│  클라이언트      │
│ curl로 요청 보냄 │
└────────┬─────────┘
         │
    HTTP 요청 전송
         ↓
┌──────────────────────────┐
│  메인 스레드             │
│  (손님 맞이하는 직원)    │
│ - 요청 받기 (accept)    │
│ - 줄에 세우기 (큐)      │
└────────┬─────────────────┘
         │
    일꾼 깨우기
         ↓
┌──────────────────────────┐
│  워커 스레드             │
│  (요리사 같은 일꾼)      │
│ - 편지 읽기 (HTTP)      │
│ - 일 파악하기 (SQL)     │
│ - 락 획득 (뮤텍스)      │
└────────┬─────────────────┘
         │
    일 지시
         ↓
┌──────────────────────────┐
│  SQL 처리기              │
│  (주방 같은 곳)          │
│ - 명령 실행              │
│ - 데이터 저장/조회       │
└────────┬─────────────────┘
         │
    결과 반환
         ↓
┌──────────────────────────┐
│  워커 스레드             │
│  - 결과를 JSON으로 변환  │
│  - 락 해제 (뮤텍스)      │
│  - 응답 보내기 (HTTP)    │
└────────┬─────────────────┘
         │
    응답 전송
         ↓
┌──────────────────┐
│  클라이언트      │
│  결과 받음       │
└──────────────────┘
```

## 🤔 자주 나오는 질문

### Q1: 스레드 풀의 워커가 4명인 이유는?
**A:** 
- 너무 적으면 (1명): 느림
- 너무 많으면 (100명): 메모리 낭비 + 느림
- 4명 정도가 균형 잡힘

### Q2: 뮤텍스가 없으면 정말 문제가 될까?
**A:**
```
뮤텍스 없이 동시에 저장:
Alice INSERT → id는 1
Bob INSERT → id도 1 (중복!)
→ 둘 다 같은 ID라서 데이터 손상
```

### Q3: JSON이 뭐예요?
**A:**
```
데이터를 정해진 형식으로 쓴 것

{
  "ok": true,
  "name": "Alice",
  "age": 20
}

→ 파이썬, 자바스크립트 등 여러 언어가 이해할 수 있음
```

### Q4: HTTP 파싱이 뭐예요?
**A:**
```
편지 받았을 때:

"POST /query HTTP/1.1
Content-Length: 42

INSERT INTO users VALUES ('Alice', 20);"

이 편지를 읽고:
- 방법: POST
- 경로: /query
- 내용: INSERT INTO users VALUES ('Alice', 20);

이렇게 분석하는 것 = 파싱
```

## ✅ 이해했는지 확인

다음 질문에 답할 수 있나요?

1. **"스레드 풀이 뭐예요?"**
   → 미리 만들어둔 일꾼 팀. 요청이 들어오면 일꾼이 처리해줌

2. **"왜 뮤텍스가 필요해요?"**
   → 여러 일꾼이 동시에 데이터를 건드리면 망하니까, 한 명씩 만 접근하게 강제

3. **"JSON이 왜 필요해요?"**
   → 결과를 클라이언트가 이해할 수 있는 형식으로 보내려고

4. **"요청-응답 흐름이 뭐예요?"**
   → 클라이언트가 요청 → 서버가 받음 → 일꾼이 처리 → 결과 응답 → 클라이언트가 받음
