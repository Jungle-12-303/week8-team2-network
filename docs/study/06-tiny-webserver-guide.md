# 06. Tiny 웹서버 이해 가이드

## Tiny의 정체
Tiny는 교육용 `iterative HTTP/1.0 web server`다. 책에서 tiny가 중요한 이유는 "실제 웹서버에서 일어나는 핵심 동작"을 짧은 코드에 압축해서 보여 주기 때문이다.

Tiny는 아래 두 가지를 처리한다.

- 정적 콘텐츠: 파일을 그대로 읽어 응답
- 동적 콘텐츠: CGI 프로그램을 실행해 그 결과를 응답

이 두 경로만 이해해도 웹서버가 "요청을 해석하고, 자원을 찾고, 응답을 만든다"는 기본 구조를 잡을 수 있다.

## Tiny가 좋은 학습 예제인 이유
Tiny는 완전한 서버가 아니다. 보안도 약하고, 기능도 많지 않고, `GET` 중심이며, 성능 최적화도 거의 없다. 그런데 바로 그 단순함 때문에 학습용으로 좋다. 필요 없는 복잡함을 걷어내고 핵심만 남겼기 때문이다.

Tiny를 통해 볼 수 있는 핵심은 아래와 같다.

- 요청 하나를 처리하는 서버의 기본 구조
- HTTP request line과 header 처리
- URI 기반 분기
- 파일 기반 응답
- CGI 기반 응답
- 에러 응답

## Tiny의 전체 처리 흐름
브라우저가 tiny에 요청을 보내면 대략 아래 흐름이 일어난다.

1. tiny가 연결을 수락한다
2. 요청 라인을 읽는다
3. method가 적절한지 확인한다
4. 헤더를 읽는다
5. URI를 해석한다
6. 정적 콘텐츠 또는 동적 콘텐츠 경로로 분기한다
7. 응답 헤더와 본문을 보낸다
8. 연결을 닫는다

이 흐름은 단순하지만 웹서버의 핵심 동작을 거의 모두 포함한다.

## request line을 읽는 이유
request line은 보통 `GET /index.html HTTP/1.0` 같은 형태다. 이 한 줄 안에 서버가 앞으로 무엇을 해야 하는지가 압축되어 있다.

- 어떤 method인가
- 어떤 자원을 원하는가
- 어떤 HTTP 버전을 쓰는가

Tiny는 교육용이므로 주로 `GET`을 중심으로 다룬다. 핵심은 "이 한 줄을 읽고 나서야 서버가 다음 행동을 결정할 수 있다"는 점이다.

## 헤더를 읽는 이유
초심자는 request line만 있으면 충분해 보일 수 있다. 하지만 HTTP는 여러 줄의 헤더를 함께 보낸다. Tiny는 복잡한 모든 헤더를 적극 활용하지 않더라도, 적어도 헤더 구간이 어디까지인지 알아야 한다.

헤더는 빈 줄이 나오면 끝난다. 따라서 `read_requesthdrs` 같은 함수는 "헤더가 필요한 정보이기 때문"만이 아니라 "요청 메시지의 경계를 정확히 처리하기 위해서"도 중요하다.

## iterative server로서의 Tiny
Tiny는 한 번에 하나의 연결만 처리하는 `iterative server`다. 즉, 연결을 하나 받아 처리하고 닫은 뒤 다음 연결을 받는다.

이 구조의 장점:
- 이해하기 쉽다
- 디버깅이 쉽다
- 요청 하나의 흐름이 선형적이다

이 구조의 한계:
- 느린 요청 하나가 전체 서버를 막는다
- 동시에 많은 요청을 처리하기 어렵다

이 한계는 Proxy Lab의 concurrency 단계에서 다시 등장한다. Tiny의 iterative 구조를 먼저 이해하면 왜 병렬 처리가 필요한지도 자연스럽게 이해된다.

## URI 해석과 분기
웹서버는 URI를 보고 어떤 자원을 줄지 판단해야 한다. Tiny는 URI를 해석해 크게 두 갈래로 나눈다.

- 정적 콘텐츠 요청
- 동적 콘텐츠 요청

이 분기가 중요한 이유는 이후 처리 방식이 완전히 달라지기 때문이다.

- 정적 콘텐츠는 파일을 찾아 읽어 보내면 된다
- 동적 콘텐츠는 프로그램을 실행해 출력을 받아야 한다

즉 URI 파싱은 단순 문자열 처리 이상의 의미를 가진다. 서버의 처리 전략을 결정하는 단계다.

## 정적 콘텐츠 처리
정적 콘텐츠 경로에서는 서버가 파일 시스템에서 파일을 찾아야 한다. 여기서 웹서버는 단순히 파일 바이트만 보내는 것이 아니라, 브라우저가 이해할 수 있도록 응답 헤더도 함께 보내야 한다.

정적 콘텐츠 처리에서 봐야 할 개념:
- 파일 존재 여부
- 읽기 권한
- 파일 크기
- MIME type
- HTTP response header

왜 중요한가:
- 파일 I/O와 네트워크 I/O가 만난다
- "응답 본문"과 "응답 메타데이터"가 함께 필요하다는 점이 드러난다

## 동적 콘텐츠와 CGI
Tiny의 동적 콘텐츠는 CGI(Common Gateway Interface) 방식으로 설명된다. 여기서 서버는 직접 모든 계산을 하는 대신, 외부 프로그램을 실행하고 그 결과를 응답으로 전달한다.

핵심 아이디어는 아래와 같다.

- 서버는 요청에서 필요한 입력을 추출한다
- 그 입력을 CGI 프로그램이 이해할 수 있는 형태로 전달한다
- CGI 프로그램은 표준 출력으로 결과를 쓴다
- 서버는 그 결과가 응답이 되도록 연결한다

이 모델이 중요한 이유는 "웹서버는 파일만 주는 프로그램이 아니라, 외부 계산 결과를 응답할 수 있는 프로그램"이라는 점을 보여 주기 때문이다.

## `adder.c`를 볼 때의 포인트
`adder.c`는 매우 작은 프로그램이지만, 아래를 함께 보여 준다.

- 쿼리 문자열 읽기
- 문자열 분해
- 숫자 변환
- HTTP 응답 헤더 출력
- HTML 본문 출력

특히 중요한 것은 `printf`가 단순 콘솔 출력이 아니라, CGI 실행 맥락에서는 결국 클라이언트로 가는 응답 일부가 될 수 있다는 점이다.

## 에러 처리
Tiny는 잘못된 method, 존재하지 않는 파일, 읽기 권한이 없는 파일 같은 실패를 HTTP 에러 응답으로 바꿔야 한다. 운영체제 에러를 그대로 보여 주는 것과 웹서버 에러 응답은 다르다.

웹서버는 실패를 아래 형태로 바꿔야 한다.

- 적절한 상태 코드
- 사람이 읽을 수 있는 짧은 설명
- 브라우저가 받아들일 수 있는 HTTP 형식

이 부분을 이해하면 "시스템 호출 실패"와 "HTTP 수준 실패"를 구분할 수 있게 된다.

## Tiny를 코드 없이 읽는 방법
코드가 길게 느껴지면 함수 이름을 따라가며 아래 질문만 붙들고 읽으면 된다.

### `main`
- 연결을 어디서 기다리는가
- 요청 하나 처리의 시작점은 어디인가

### `doit`
- 요청 라인을 어디서 읽는가
- method와 URI를 어디서 검사하는가
- 정적/동적 분기는 어디서 일어나는가

### `read_requesthdrs`
- 헤더 끝을 무엇으로 판단하는가

### `parse_uri`
- 어떤 기준으로 콘텐츠 종류를 나누는가

### `serve_static`
- 응답 헤더와 본문을 어떻게 구분해 보내는가

### `serve_dynamic`
- CGI 프로그램과 서버의 책임은 어떻게 나뉘는가

### `clienterror`
- 실패를 어떻게 HTTP 형식으로 표현하는가

## Tiny와 Proxy를 연결해서 보기
Tiny를 제대로 이해하면 proxy에서 아래가 쉬워진다.

- HTTP 요청/응답 구조가 이미 익숙하다
- 브라우저와 서버 사이에 어떤 텍스트가 오가는지 안다
- 응답 헤더의 중요성을 이해한다
- iterative 구조의 한계를 알고 있다

즉 proxy는 "전혀 새로운 과제"가 아니라, tiny를 바탕으로 책임이 늘어난 과제라는 점이 보인다.
