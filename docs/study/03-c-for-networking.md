# 03. 네트워크 프로그래밍을 위한 C 문법

## 왜 네트워크 코드가 C 초심자에게 어렵게 보이는가
네트워크 코드는 문법 하나가 어려운 것이 아니라, 여러 층위의 개념이 한꺼번에 등장해서 어렵다.

- 문자열과 버퍼
- 포인터와 배열
- 구조체
- 시스템 호출
- 에러 처리
- 파일 디스크립터
- 경우에 따라 프로세스와 스레드

즉 "C 문법"과 "운영체제 개념"과 "네트워크 개념"이 한 줄에 함께 등장한다. 그래서 읽는 순서를 정해 두는 것이 중요하다. 먼저 함수가 무엇을 하려는지 보고, 그 다음 인자 타입과 포인터를 본다.

## 문자열과 버퍼
HTTP는 텍스트 기반 프로토콜이므로 tiny와 proxy 코드에는 문자열 버퍼가 많이 등장한다. C에서 문자열은 널 문자(`\0`)로 끝나는 문자 배열이다. 네트워크 데이터는 처음에는 그저 바이트들의 모음으로 들어오지만, HTTP 요청 라인과 헤더는 문자열처럼 다룰 수 있다.

왜 중요한가:
- 요청 라인과 헤더를 저장하고 파싱해야 한다.
- 응답 헤더도 문자열로 조합해서 보내게 된다.

tiny/proxy에서 어디 쓰이는가:
- request line 저장
- URI, method, version 분리
- 응답 헤더 작성

자기점검 질문: "바이트 스트림"을 문자열처럼 읽을 수 있는 경우와 없는 경우는 무엇이 다를까?

## 배열과 포인터
`char buf[MAXLINE]`는 실제 메모리 공간이 스택에 잡힌 배열이고, `char *buf`는 어떤 메모리 위치를 가리키는 포인터다. 네트워크 코드에서는 둘 다 자주 보인다.

배열은 버퍼 공간을 직접 확보할 때 편하고, 포인터는 그 버퍼 안의 특정 위치를 가리키거나 함수 간 전달을 할 때 편하다.

왜 중요한가:
- 버퍼를 읽고, 일부를 가리키고, 내용을 조합하는 작업이 많기 때문이다.

tiny/proxy에서 어디 쓰이는가:
- 헤더 줄을 읽은 뒤 특정 문자 위치를 찾기
- URI 안에서 쿼리 문자열 시작 위치 찾기
- 응답 버퍼 뒤에 문자열을 이어 붙이기

자기점검 질문: 배열과 포인터를 혼동하면 버퍼 길이 계산에서 어떤 실수를 하게 될까?

## 문자열 함수
`strlen`, `strcmp`, `strcpy`, `strcat`, `strchr`, `sprintf` 같은 함수는 HTTP 문자열 처리에 자주 등장한다. 하지만 편리한 만큼 길이와 종료 문자를 의식해야 한다.

왜 중요한가:
- C는 자동으로 문자열 길이를 보호해 주지 않는다.
- 네트워크 입력은 외부 입력이므로 버퍼 크기 감각이 더 중요하다.

tiny/proxy에서 어디 쓰이는가:
- request line 파싱
- file type 문자열 조합
- 에러 응답 메시지 구성

자기점검 질문: 문자열 함수가 많은 C 코드에서 "코드는 짧은데 위험도는 높다"는 말이 왜 나올까?

## 구조체
소켓 주소 정보는 여러 필드를 함께 전달해야 하므로 구조체가 쓰인다. `struct sockaddr`, `struct sockaddr_storage`, `struct addrinfo`는 모두 "주소와 관련된 여러 정보를 한 덩어리로 다루기 위한 타입"이라고 이해하면 된다.

왜 중요한가:
- 소켓 함수의 인자는 대부분 주소 구조체를 받는다.
- 한 번에 하나의 숫자만 전달해서는 충분하지 않다.

tiny/proxy에서 어디 쓰이는가:
- `accept`로 클라이언트 주소 받기
- `getaddrinfo`로 원서버 연결 정보 받기

자기점검 질문: 주소 정보를 구조체로 묶는 것이 함수 인터페이스에 어떤 장점을 줄까?

## 타입 캐스팅이 자주 보이는 이유
CSAPP 코드에서 `(SA *)&clientaddr` 같은 캐스팅이 자주 보인다. 여기서 `SA`는 `struct sockaddr`의 typedef다. 여러 주소 구조체를 공통 인터페이스에 넘기기 위해 캐스팅이 사용된다.

왜 중요한가: 코드를 읽을 때 "형변환이 복잡해서 무서운 것"이 아니라 "공통 인터페이스에 맞추는 것"이라고 이해하면 부담이 줄어든다.

tiny/proxy에서 어디 쓰이는가:
- `Accept`, `Getnameinfo` 호출

자기점검 질문: 왜 `sockaddr_storage`를 `sockaddr *`로 넘기는가?

## 함수 원형과 래퍼 함수
CSAPP는 원시 시스템 호출 위에 래퍼를 얹어 사용한다. 예를 들어 `accept` 대신 `Accept`, `open_listenfd` 대신 `Open_listenfd`, `rio_readlineb` 대신 `Rio_readlineb`를 사용한다.

왜 중요한가:
- 에러 처리를 통일할 수 있다.
- 코드의 중심 흐름이 더 잘 보인다.

tiny/proxy에서 어디 쓰이는가:
- 거의 모든 네트워크 I/O

자기점검 질문: 학습 단계에서는 원시 시스템 호출보다 래퍼 함수가 왜 더 읽기 쉬운가?

## file descriptor
`int fd` 하나가 파일일 수도 있고 소켓일 수도 있다는 감각이 중요하다. 운영체제는 열린 I/O 대상을 정수로 식별하고, 프로세스는 그 정수를 통해 읽기/쓰기/닫기를 요청한다.

왜 중요한가:
- 네트워크도 결국 I/O라는 사실이 드러난다.
- 같은 `Read`, `Write`, `Close` 패턴이 왜 소켓 코드에도 나오는지 이해할 수 있다.

tiny/proxy에서 어디 쓰이는가:
- `listenfd`
- `connfd`
- `clientfd`

자기점검 질문: 리스닝 소켓과 연결 소켓은 둘 다 file descriptor인데 왜 역할은 전혀 다를까?

## rio_t와 Robust I/O
CSAPP의 `rio_t`는 내부 버퍼 상태를 관리하면서 줄 단위 읽기와 안정적인 읽기를 돕는다. HTTP처럼 줄 경계가 중요한 프로토콜을 다룰 때 특히 유용하다.

왜 중요한가:
- 단순 `read`는 한 번에 원하는 만큼 딱 떨어지게 읽는다는 보장이 없다.
- request line과 header는 줄 단위로 읽는 편이 자연스럽다.

tiny/proxy에서 어디 쓰이는가:
- `Rio_readlineb`로 request line 읽기
- 헤더를 빈 줄이 나올 때까지 읽기

자기점검 질문: `Rio_readlineb`가 없으면 tiny의 요청 헤더 읽기 코드는 어떤 점에서 더 불편해질까?

## fork, execve, CGI
tiny의 동적 콘텐츠는 CGI 프로그램 실행과 연결된다. 서버는 요청을 읽은 뒤 직접 모든 계산을 하지 않고, 별도 프로그램을 실행해 그 출력을 응답으로 사용할 수 있다.

왜 중요한가:
- 네트워크 코드와 프로세스 제어가 어떻게 이어지는지 볼 수 있다.
- 웹서버가 반드시 "모든 로직을 한 파일 안에서" 처리하는 것이 아니라는 점을 이해하게 된다.

tiny/proxy에서 어디 쓰이는가:
- `cgi-bin/adder.c` 같은 프로그램의 역할 이해

자기점검 질문: 서버가 외부 프로그램을 실행해 응답을 만들면 어떤 유연성이 생길까?

## 스레드와 pthread
Proxy Lab의 병렬 처리 단계에서는 한 요청이 오래 걸려도 다른 요청을 처리할 수 있어야 한다. 이때 스레드가 등장한다.

왜 중요한가:
- concurrent server 개념을 실제 구현 선택과 연결할 수 있다.
- 동시성이 들어오면 공유 자원과 동기화라는 새 문제도 함께 등장한다.

tiny/proxy에서 어디 쓰이는가:
- `Pthread_create`
- `Pthread_detach`
- 캐시 공유 설계와 연결

자기점검 질문: 스레드가 들어오면 요청 처리 속도만 좋아지는 것이 아니라 설계가 어려워지는 이유는 무엇인가?

## 이 문서를 읽고 나서 할 일
1. 함수 이름만 보지 말고 인자의 타입과 역할을 같이 읽는다.
2. 문자열 버퍼, 구조체, file descriptor, `rio_t`가 코드에 나오면 각각 무슨 층위의 문제를 다루는지 구분한다.
3. tiny를 읽을 때는 "HTTP 해석"에 집중하고, proxy를 읽을 때는 "양쪽 연결 관리"에 집중한다.
