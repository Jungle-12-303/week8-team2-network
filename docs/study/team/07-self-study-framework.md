# 07. CSAPP 11장 자기주도 학습 프레임워크

이 문서는 11.6c, 11.7, 11.9, 11.10, 11.11의 정답이나 해설을 제공하지 않는다. 대신 문제를 풀기 전에 무엇을 먼저 확인해야 하는지, 무엇을 설명할 수 있어야 하는지를 정리한다.

## 사용 방법
1. 문제를 먼저 읽는다.
2. 아래 "다시 볼 키워드"를 기준으로 책과 코드로 돌아간다.
3. "내가 직접 설명해볼 질문"에 답해 본다.
4. 그 다음에만 문제 풀이를 시작한다.

## 11.6c
이 문제는 무엇을 묻는가: 클라이언트/서버 연결 또는 주소 처리 흐름을 정확히 이해했는지 점검하는 문제로 읽어야 한다.

답을 쓰기 전에 다시 볼 키워드:
- client-server model
- socket address
- hostname, IP address, port
- `open_clientfd`, `open_listenfd`

내가 직접 설명해볼 질문:
- 클라이언트는 연결 전에 어떤 정보를 알아야 하는가?
- 서버는 왜 포트를 열고 기다려야 하는가?
- 이름과 주소는 어떤 단계에서 연결되는가?

tiny/proxy 구현과 연결되는 이유:
- tiny는 서버 쪽 흐름을 이해해야 하고, proxy는 서버와 클라이언트 양쪽 흐름을 동시에 이해해야 한다.

## 11.7
이 문제는 무엇을 묻는가: 소켓 API 호출 순서와 각 호출의 책임을 구분할 수 있는지 점검하는 문제로 읽어야 한다.

답을 쓰기 전에 다시 볼 키워드:
- `socket`
- `bind`
- `listen`
- `accept`
- `connect`

내가 직접 설명해볼 질문:
- 서버가 `accept` 전에 끝내야 하는 준비는 무엇인가?
- 클라이언트 흐름과 서버 흐름은 어디서 갈라지는가?
- 어떤 호출이 "대기"를 만든다고 볼 수 있는가?

tiny/proxy 구현과 연결되는 이유:
- tiny는 기본 서버 흐름을 그대로 사용하고, proxy는 `accept` 뒤에 다시 `connect`를 해야 한다.

## 11.9
이 문제는 무엇을 묻는가: HTTP 또는 애플리케이션 계층 요청 처리의 의미를 이해했는지 점검하는 문제로 읽어야 한다.

답을 쓰기 전에 다시 볼 키워드:
- request line
- headers
- URI
- GET

내가 직접 설명해볼 질문:
- 서버는 요청의 어느 부분을 먼저 읽는가?
- URI만 보고 알 수 있는 정보는 무엇인가?
- 헤더는 왜 한 줄씩 읽는가?

tiny/proxy 구현과 연결되는 이유:
- tiny는 HTTP 요청을 해석해야 하고, proxy는 해석한 요청을 원서버가 이해할 수 있는 형태로 다시 전달해야 한다.

## 11.10
이 문제는 무엇을 묻는가: 동적 콘텐츠 또는 프로세스 실행 맥락이 어떻게 HTTP 응답과 연결되는지 점검하는 문제로 읽어야 한다.

답을 쓰기 전에 다시 볼 키워드:
- CGI
- environment variable
- `fork`
- `execve`

내가 직접 설명해볼 질문:
- 서버는 CGI 프로그램에 입력을 어떻게 넘길까?
- CGI 프로그램의 출력은 왜 응답이 될 수 있을까?
- 정적 콘텐츠 처리와 동적 콘텐츠 처리는 어디서 분기되는가?

tiny/proxy 구현과 연결되는 이유:
- `adder.c`를 이해하면 tiny가 파일 서버를 넘어서 "프로그램 실행 결과를 응답하는 서버"라는 점이 보인다.

## 11.11
이 문제는 무엇을 묻는가: 동시성, 공유 자원, 또는 네트워크 서버의 성능/구조 문제를 이해했는지 점검하는 문제로 읽어야 한다.

답을 쓰기 전에 다시 볼 키워드:
- iterative server
- concurrent server
- thread
- shared state

내가 직접 설명해볼 질문:
- 느린 요청 하나가 서버 전체를 막는 구조는 어떤 구조인가?
- 스레드를 사용하면 무엇이 좋아지고 무엇이 어려워지는가?
- 캐시가 들어오면 공유 자원 문제는 왜 생길 수 있는가?

tiny/proxy 구현과 연결되는 이유:
- Proxy Lab의 핵심 확장 포인트가 바로 동시성과 캐시다.

## 마지막 점검
- 이 문제들을 풀기 전에 tiny와 proxy 중 어느 쪽에 더 직접 연결되는지 구분해 보자.
- 답을 작성한 뒤에는 "책 설명"이 아니라 "내 구현 선택"과 연결해서 다시 말해 보자.
