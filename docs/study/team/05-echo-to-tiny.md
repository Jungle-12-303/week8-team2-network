# 05. Echo Server에서 Tiny로

## 왜 echo server를 먼저 떠올려야 하는가
echo server는 네트워크 요청/응답의 최소 형태다. 서버는 연결을 받고, 데이터를 읽고, 다시 데이터를 보낸다. 여기에 HTTP도 없고 파일도 없고 브라우저도 꼭 필요하지 않다. 하지만 네트워크 서버의 뼈대는 이미 모두 들어 있다.

즉 echo server는 "서버의 몸통"을 보여 주고, tiny는 그 몸통 위에 "웹서버의 의미"를 덧붙인 예제라고 볼 수 있다.

## echo server의 관점
echo server의 질문은 단순하다.

- 누가 연결했는가
- 어떤 바이트가 들어왔는가
- 그 바이트를 다시 어떻게 돌려줄 것인가

echo server는 요청의 의미를 거의 해석하지 않는다. 따라서 프로토콜 처리보다는 연결과 입출력의 리듬을 익히는 데 좋다.

## tiny로 넘어가면서 추가되는 것
tiny는 echo server가 하던 "읽고 쓰기"를 그대로 유지하되, 그 사이에 해석과 분기를 넣는다.

- 요청 라인을 읽는다
- method를 검사한다
- URI를 해석한다
- 헤더를 읽는다
- 파일 요청인지 CGI 요청인지 구분한다
- 적절한 HTTP 응답을 작성한다

즉 tiny는 "요청 바이트"를 "의미 있는 웹 요청"으로 해석한다.

## echo와 tiny의 본질적 차이

### echo
- 데이터 자체에 큰 의미를 부여하지 않는다
- 응답이 입력에 거의 직접적으로 대응한다
- 프로토콜 해석이 최소다

### tiny
- 입력을 HTTP 요청으로 해석한다
- 해석 결과에 따라 응답 경로가 갈린다
- 파일 시스템, MIME 타입, CGI, 에러 응답이 함께 들어온다

## tiny의 핵심 함수들을 "책임" 기준으로 읽기

### `main`
반복적으로 연결을 받고 요청 처리 함수를 호출한다. 네트워크 서버의 바깥쪽 틀이다.

### `doit`
요청 하나를 끝까지 처리하는 핵심 함수다. method 검사, URI 해석, 파일/CGI 분기, 에러 처리 등 대부분의 논리가 이 함수 주변에 모인다.

### `read_requesthdrs`
헤더를 어디까지 읽어야 하는지 담당한다. HTTP 요청은 빈 줄이 나와야 헤더가 끝난다.

### `parse_uri`
URI가 정적 콘텐츠를 가리키는지 동적 콘텐츠를 가리키는지 나누고, 이후 처리에 필요한 문자열을 준비한다.

### `serve_static`
파일을 읽어 응답하는 경로다.

### `serve_dynamic`
CGI 프로그램을 실행해 그 출력을 응답으로 사용하는 경로다.

### `clienterror`
실패 상황을 HTTP 에러 응답으로 바꾸는 함수다.

## `adder.c`를 왜 꼭 봐야 하는가
`adder.c`는 tiny가 단순 파일 서버가 아니라 "프로그램 실행 결과를 돌려주는 서버"도 될 수 있음을 보여 준다. 이 예제를 통해 아래 개념이 연결된다.

- 환경 변수
- 쿼리 문자열
- CGI
- 표준 출력과 HTTP 응답의 관계

즉 `adder.c`는 작은 예제지만, 웹서버가 외부 프로그램과 협력해 동적 응답을 만들 수 있다는 핵심 개념을 담고 있다.

## echo를 이해하면 tiny의 어디가 쉬워지는가
- 연결을 받고 닫는 흐름이 덜 낯설어진다
- `read`와 `write`가 서버 코드의 중심이라는 점이 익숙해진다
- tiny에서 새로 어려운 부분이 "네트워크"보다 "HTTP 해석"이라는 점이 보인다

## tiny를 이해하면 proxy의 어디가 쉬워지는가
- HTTP 요청/응답의 구조가 익숙해진다
- request line, header, body 구분이 된다
- 서버가 응답을 직접 만든다는 감각이 생긴다
- 이후 proxy에서 "응답을 만들기보다 전달한다"는 차이가 더 잘 보인다
