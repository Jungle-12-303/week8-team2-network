# 08. Proxy Lab 이해 가이드

## 프록시 서버를 큰 그림에서 보기
프록시 서버는 브라우저와 웹서버 사이에 있는 중간자다. 하지만 "중간에 서 있다"는 표현만으로는 충분하지 않다. 프록시는 실제로 두 방향의 네트워크 역할을 동시에 수행한다.

- 브라우저가 볼 때: 프록시는 서버다
- 원서버가 볼 때: 프록시는 클라이언트다

이 점이 tiny와 가장 큰 차이다. tiny는 서버 역할만 보면 되지만, proxy는 서버 흐름과 클라이언트 흐름을 한 프로그램 안에서 모두 이해해야 한다.

## 브라우저 - 프록시 - 웹서버 흐름
브라우저가 프록시를 통해 웹페이지를 요청한다고 가정하면 전체 흐름은 아래처럼 진행된다.

1. 브라우저가 프록시에 연결한다
2. 프록시가 브라우저의 HTTP 요청을 읽는다
3. 프록시가 요청에서 원서버 정보를 파악한다
4. 프록시가 원서버에 새 연결을 연다
5. 프록시가 원서버에 요청을 보낸다
6. 원서버가 응답을 보낸다
7. 프록시가 그 응답을 읽는다
8. 프록시가 브라우저에게 응답을 전달한다

핵심은 proxy가 단순히 바이트를 한쪽에서 다른 쪽으로 복사하는 프로그램이 아니라, 연결과 요청의 의미를 모두 다뤄야 한다는 점이다.

## 왜 Tiny 다음에 Proxy를 공부하는가
Tiny를 먼저 공부하는 이유는 HTTP 요청/응답의 구조를 먼저 익히기 위해서다. Proxy는 tiny 위에 아래 개념이 추가된 버전이라고 보면 된다.

- 서버 역할 + 클라이언트 역할
- 요청 중계
- 응답 중계
- 동시성
- 캐시

즉 tiny가 HTTP 서버의 최소형이라면, proxy는 네트워크 프로그래밍 관점에서 더 시스템적인 고민이 들어간 확장 과제다.

## 로컬 과제 구조
`../webproxy_lab_docker/webproxy-lab`에서 확인할 수 있는 핵심 파일은 아래와 같다.

- `tiny/tiny.c`: tiny 웹서버 구현
- `tiny/cgi-bin/adder.c`: CGI 예제
- `proxy.c`: 프록시 서버 구현 대상
- `driver.sh`: Proxy Lab 동작 검증 스크립트

이 구성 자체가 학습 순서를 알려 준다. tiny를 먼저 이해하고, 그 다음 proxy를 구현해 보라는 구조다.

## Proxy Lab의 핵심 기능
과제 설명의 핵심 기능은 세 가지다.

- 순차적 요청 처리
- 병렬적 요청 처리
- 요청 캐시

이 세 가지는 단지 기능 목록이 아니라 학습 단계다. 한 번에 전부 해결하려 하지 말고 단계별로 분리해서 봐야 한다.

## 1. 순차 처리의 의미
순차 처리 프록시는 한 요청을 끝까지 처리하고 나서 다음 요청으로 넘어간다. 이 단계의 목적은 프록시의 최소 동작을 완성하는 것이다.

즉 아래를 할 수 있어야 한다.

- 브라우저 요청을 받는다
- 요청에서 원서버 정보를 뽑는다
- 원서버에 연결한다
- 요청을 전달한다
- 응답을 읽는다
- 브라우저에 다시 보낸다

이 단계가 되는지 보는 것이 드라이버의 `Basic` 테스트와 연결된다.

## 2. 병렬 처리의 의미
순차 처리 프록시는 구조가 단순하지만, 느린 요청 하나가 전체 프록시를 막는 문제가 있다. Proxy Lab은 바로 이 약점을 드러내기 위해 응답하지 않는 서버(`nop-server`)를 이용한 테스트를 제공한다.

이 단계에서 이해해야 하는 질문은 아래와 같다.

- 왜 요청 하나가 다른 요청을 막는가
- "동시에 처리한다"는 것은 소켓과 스레드 수준에서 무슨 뜻인가
- 연결 하나를 처리하는 일을 독립 작업 단위로 볼 수 있는가

이 단계가 되는지 보는 것이 드라이버의 `Concurrency` 테스트다.

## 3. 캐시의 의미
캐시는 이전에 받아 둔 응답을 메모리 등에 보관해 두었다가, 같은 요청이 들어오면 원서버까지 다시 가지 않고 응답하는 기능이다.

캐시가 중요한 이유는 단순히 "속도" 때문만이 아니다.

- 응답 지연 감소
- 원서버 부하 감소
- 네트워크 트래픽 감소
- 원서버가 없어도 일부 응답 가능

드라이버의 `Cache` 테스트는 실제로 tiny를 종료한 뒤에도 캐시된 객체를 응답할 수 있는지 확인한다. 이 테스트를 보면 캐시가 단순 부가 기능이 아니라 프록시의 핵심 확장임을 알 수 있다.

## tiny와 proxy의 차이

### tiny
- 브라우저 요청을 직접 받는다
- 요청을 해석한다
- 파일을 읽거나 CGI를 실행한다
- 응답을 직접 만든다

### proxy
- 브라우저 요청을 받는다
- 요청이 향해야 할 원서버를 찾는다
- 원서버에 새 연결을 만든다
- 요청과 응답을 중계한다
- 동시에 여러 요청을 다룰 수 있어야 한다
- 같은 객체를 다시 요청받으면 캐시를 고려해야 한다

tiny는 "응답을 만드는 서버"이고, proxy는 "응답을 가져와 전달하는 서버"라고 요약할 수 있다. 하지만 proxy가 더 쉬운 것은 아니다. 오히려 연결과 상태 관리가 늘어나기 때문에 더 어렵다.

## driver 기준으로 보는 프록시 과제
로컬 `driver.sh`는 크게 세 가지를 검사한다.

### Basic
프록시를 통해 가져온 파일과 tiny에서 직접 가져온 파일이 같은지 본다.

이 테스트가 의미하는 것:
- 프록시가 데이터를 빼먹지 않는가
- 바이너리 파일과 텍스트 파일을 모두 제대로 전달하는가
- 최소한의 요청/응답 중계가 맞게 동작하는가

### Concurrency
응답하지 않는 서버 요청이 하나 걸려 있어도, 다른 tiny 요청을 계속 처리할 수 있는지 본다.

이 테스트가 의미하는 것:
- 프록시가 한 요청 때문에 전체적으로 멈추지 않는가
- 병렬 처리 구조가 실제로 살아 있는가

### Cache
이전에 받아 둔 객체를 tiny가 꺼진 뒤에도 프록시가 돌려줄 수 있는지 본다.

이 테스트가 의미하는 것:
- 응답을 단순 전달만 하는 것이 아니라 저장하고 재사용하는가
- 원서버 없이도 캐시된 응답을 제공할 수 있는가

## 프록시를 이해할 때 새로 생기는 개념

### 두 개의 연결
프록시는 브라우저와의 연결 하나, 원서버와의 연결 하나를 동시에 다룬다.

### 요청 재구성
브라우저가 보낸 요청과 원서버에 그대로 보낼 요청이 완전히 같다고 가정하면 안 된다. 요청 라인, 헤더, 호스트 정보 등을 어떻게 다룰지 생각해야 한다.

### 병렬성
요청 하나가 오래 걸릴 수 있다는 사실을 설계에 반영해야 한다.

### 공유 캐시
캐시가 들어오면 단순한 네트워크 중계 프로그램이 아니라, 공유 상태를 관리하는 프로그램이 된다.

## Tiny와 Proxy를 연결해서 이해하기
Tiny는 "HTTP 서버가 어떻게 응답을 만드는가"를 보여 준다. Proxy는 "그 응답이 브라우저까지 도착하기 전에 중간자에서 어떤 처리가 가능해지는가"를 보여 준다.

따라서 tiny를 이해하고 나면 proxy에서 아래 질문이 자연스럽게 생긴다.

- 브라우저 요청은 누가 먼저 읽는가
- 응답은 누가 먼저 받는가
- 중간에서 무엇을 바꿀 수 있는가
- 무엇을 저장해 둘 수 있는가

## 이 문서를 읽고 나서 설명할 수 있어야 하는 것
1. proxy가 tiny보다 쉬워 보이지만 실제로는 더 복잡한 이유는 무엇인가?
2. 프록시가 서버이면서 동시에 클라이언트라는 말의 정확한 뜻은 무엇인가?
3. `Basic`, `Concurrency`, `Cache`가 각각 어떤 구조적 책임을 검증하는가?
