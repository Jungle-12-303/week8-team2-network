# 02. 핵심 개념 정리

## 먼저 큰 그림: 네트워크 프로그래밍은 무엇을 하는가
네트워크 프로그래밍은 결국 "내 프로세스와 다른 컴퓨터의 프로세스가 어떻게 데이터를 주고받는가"를 코드로 다루는 일이다. CSAPP 11장은 이 문제를 운영체제와 시스템 프로그래밍 관점에서 설명한다. 브라우저나 앱에서 요청을 보낸다는 말도, 서버가 응답한다는 말도 실제 코드로 내려오면 결국 아래 세 질문으로 바뀐다.

1. 누구와 통신할 것인가
2. 어떻게 연결할 것인가
3. 연결 뒤에 어떤 바이트를 읽고 쓸 것인가

이 세 질문을 푸는 재료가 주소, 포트, 소켓, TCP, DNS, HTTP다.

## 클라이언트-서버 모델
클라이언트는 요청을 시작하는 쪽이고, 서버는 요청을 기다리는 쪽이다. 여기서 중요한 것은 "누가 먼저 연결을 시작하느냐"다. 서버는 특정 포트에서 대기하고, 클라이언트는 그 포트로 접속한다.

왜 중요한가: 이 모델을 이해하지 못하면 `listen`, `accept`, `connect`의 역할이 섞여 보인다.

tiny/proxy에서 어디 쓰이는가:
- tiny는 전형적인 서버다.
- 브라우저는 tiny의 클라이언트다.
- proxy는 브라우저 앞에서는 서버이고, 원서버 앞에서는 클라이언트다.

자기점검 질문: 같은 프로그램이 상황에 따라 서버 역할과 클라이언트 역할을 동시에 가질 수 있는 이유는 무엇인가?

## 프로세스와 네트워크 연결
네트워크는 "컴퓨터 대 컴퓨터" 수준으로만 보면 너무 넓다. 실제 프로그래밍에서는 한 컴퓨터 안의 특정 프로세스가 다른 컴퓨터 안의 특정 프로세스와 통신한다. 이때 특정 프로세스를 가리키는 식별에 포트가 필요하다.

왜 중요한가: IP 주소만으로는 어느 프로그램과 통신할지 정해지지 않는다.

tiny/proxy에서 어디 쓰이는가: tiny를 8000번 포트에 띄우고, proxy를 다른 포트에 띄우는 이유가 여기 있다.

자기점검 질문: 같은 컴퓨터에서 tiny와 proxy를 동시에 띄울 수 있는 이유는 무엇인가?

## BSD 소켓
BSD 소켓은 네트워크 통신을 파일 I/O처럼 다루게 해 주는 운영체제 인터페이스다. 운영체제는 소켓을 통해 네트워크 연결을 추상화하고, 프로그램은 그 소켓에 대해 읽기/쓰기/닫기 같은 동작을 수행한다.

교재 관점에서 중요한 포인트는 "소켓은 추상화"라는 점이다. 네트워크 카드, 패킷 분할, 물리적인 선로를 직접 다루지 않고도 네트워크 통신 프로그램을 작성할 수 있는 이유가 여기에 있다.

왜 중요한가:
- 네트워크 코드의 진입점이기 때문이다.
- file descriptor 모델과 연결되기 때문이다.
- tiny와 proxy의 거의 모든 흐름이 결국 소켓 위에서 일어난다.

tiny/proxy에서 어디 쓰이는가:
- 서버는 리스닝 소켓을 만든다.
- 연결이 들어오면 연결 소켓을 얻는다.
- proxy는 원서버 연결용 소켓도 추가로 연다.

자기점검 질문: 소켓 추상화가 없다면 애플리케이션 프로그래머는 어떤 복잡함을 직접 떠안아야 할까?

## IP 주소
IP 주소는 네트워크에서 호스트를 식별하는 주소다. 하지만 애플리케이션 코드 입장에서는 숫자 IP 자체보다 "연결 대상 호스트를 식별하는 정보"로 보는 편이 실용적이다.

IP는 "어느 컴퓨터인가"를 알려 주고, 포트는 "그 컴퓨터 안의 어느 서비스인가"를 알려 준다. 둘이 합쳐져야 실제 통신 대상을 정할 수 있다.

왜 중요한가: 브라우저가 요청하는 대상 서버를 proxy가 찾으려면 결국 호스트와 포트를 알아야 한다.

tiny/proxy에서 어디 쓰이는가:
- tiny는 자신의 포트에 바인딩된다.
- proxy는 브라우저 요청에 들어 있는 원서버 호스트와 포트를 바탕으로 연결을 만든다.

자기점검 질문: `localhost:8000`이라는 표현에 이미 어떤 두 종류의 정보가 들어 있는가?

## 포트
포트는 한 호스트 안에서 특정 프로세스 또는 서비스를 구분하는 번호다. 흔히 "문 번호" 비슷하게 설명하지만, 더 정확히는 "운영체제가 네트워크 데이터를 어느 소켓으로 넘길지 결정하는 식별자"라고 보면 된다.

왜 중요한가: 서버는 포트에 `bind`해서 기다리고, 클라이언트는 그 포트로 `connect`한다.

tiny/proxy에서 어디 쓰이는가:
- tiny와 proxy는 서로 다른 포트에서 실행되어야 한다.
- 드라이버가 free port를 찾아 두 프로그램을 각각 띄우는 이유도 포트 충돌을 피하기 위해서다.

자기점검 질문: 서버 코드를 실행했는데 "Address already in use"가 뜬다면 무엇을 먼저 의심해야 할까?

## DNS
DNS는 사람이 읽기 쉬운 호스트 이름을 IP 주소로 바꾸는 시스템이다. 프로그래머 입장에서는 "문자열 호스트 이름을 실제 연결 가능한 주소 정보로 바꾸는 단계"라고 이해하면 된다.

교재에서 중요한 포인트는 이름 해석이 네트워크 프로그래밍의 일부라는 점이다. 사용자는 URL을 입력하지만 프로그램은 결국 주소 정보를 얻어야 연결을 만들 수 있다.

왜 중요한가: proxy는 브라우저가 보낸 서버 이름을 원서버 연결에 사용할 수 있는 정보로 바꿔야 한다.

tiny/proxy에서 어디 쓰이는가:
- `Open_clientfd`, `Getaddrinfo` 계열 함수가 이 역할과 연결된다.
- tiny는 주로 서버 역할이라 이름 해석보다 바인딩이 중심이지만, proxy는 원서버 연결 때문에 이름 해석이 더 중요하다.

자기점검 질문: 브라우저가 `http://example.com/index.html`을 요청하면 proxy는 연결 전에 최소 무엇을 추출해야 할까?

## file descriptor
file descriptor는 운영체제가 열린 I/O 객체에 붙이는 정수 핸들이다. 파일, 소켓, 파이프 모두 file descriptor로 다룰 수 있다. 이 점이 네트워크 프로그래밍을 시스템 프로그래밍과 자연스럽게 연결해 준다.

교재에서 이 개념이 중요한 이유는 네트워크도 결국 I/O라는 사실을 강조하기 때문이다. 그래서 소켓에 대해서도 `Read`, `Write`, `Close` 같은 흐름이 등장한다.

왜 중요한가:
- 소켓이 특별한 마법 객체가 아니라 I/O 대상이라는 점을 이해해야 한다.
- `listenfd`, `connfd`, `clientfd` 같은 이름의 의미를 이해할 수 있다.

tiny/proxy에서 어디 쓰이는가:
- `listenfd`: 서버가 연결을 기다리는 소켓
- `connfd`: 실제 클라이언트와 통신하는 연결 소켓
- `clientfd`: proxy가 원서버에 연결할 때 사용하는 클라이언트 소켓

자기점검 질문: 리스닝 소켓과 연결 소켓을 분리하는 설계는 서버에 어떤 이점을 줄까?

## TCP
TCP는 연결 지향적이고 신뢰성 있는 바이트 스트림 서비스를 제공한다. 여기서 중요한 표현은 "바이트 스트림"이다. TCP는 메시지 경계를 자동으로 보존해 주는 것이 아니라, 순서 있는 바이트 흐름을 제공한다.

따라서 HTTP 요청도 실제로는 "한 덩어리 메시지"가 아니라 TCP 위를 흘러가는 텍스트 바이트들의 연속이다. 서버는 그 바이트 흐름에서 줄 경계와 헤더 종료 지점을 스스로 찾아야 한다.

왜 중요한가:
- tiny와 proxy는 모두 TCP 연결 위에서 HTTP를 읽고 쓴다.
- `Rio_readlineb` 같은 줄 단위 읽기 함수가 왜 필요한지 이해하게 된다.

tiny/proxy에서 어디 쓰이는가:
- 요청 라인 읽기
- 헤더 읽기
- 응답 전달

자기점검 질문: TCP가 메시지 단위가 아니라 바이트 스트림이라는 점이 HTTP 파싱 코드에 어떤 요구를 만들까?

## UDP
UDP는 연결이 없고, 전달 보장도 없는 데이터그램 서비스다. 이번 과제에서는 직접 사용하지 않지만, 비교 대상으로서 중요하다.

TCP는 신뢰성과 순서 보장을 제공하는 대신 관리 비용이 있다. UDP는 단순하지만 애플리케이션이 더 많은 책임을 져야 한다. 웹서버 과제가 TCP를 사용하는 이유는 HTTP 요청/응답이 신뢰성 있는 스트림 위에서 다뤄지는 것이 자연스럽기 때문이다.

왜 중요한가: "왜 하필 TCP인가"를 설명할 수 있어야 네트워크 프로그래밍 선택의 의미가 보인다.

tiny/proxy에서 어디 쓰이는가: 직접 쓰이지는 않지만, HTTP와 서버 과제의 전제를 설명할 때 필요하다.

자기점검 질문: 만약 tiny를 UDP 위에서 만든다면 요청/응답의 어떤 부분이 더 어려워질까?

## HTTP
HTTP는 애플리케이션 계층의 요청/응답 프로토콜이다. 브라우저와 웹서버는 HTTP 메시지를 주고받는다. 이 메시지는 요청 라인, 헤더, 그리고 필요하다면 본문으로 이루어진다.

이번 학습에서 중요한 것은 HTTP 전체 스펙을 외우는 것이 아니다. tiny와 proxy 구현에 필요한 최소 구조를 이해하는 것이다.

- 요청 라인: method, URI, version
- 헤더: 부가 정보
- 빈 줄: 헤더 끝
- 본문: 필요할 때만 존재

왜 중요한가:
- tiny는 HTTP 요청을 해석해야 한다.
- proxy는 받은 HTTP 요청과 응답을 중계해야 한다.

tiny/proxy에서 어디 쓰이는가:
- tiny는 method를 검사하고 URI를 파싱한다.
- proxy는 요청을 읽어 원서버로 전달하고 응답을 다시 브라우저에 넘긴다.

자기점검 질문: echo server는 바이트만 돌려줘도 되는데 tiny는 왜 request line을 파싱해야 할까?

## HTTP Method
HTTP method는 클라이언트가 자원에 대해 어떤 동작을 원하고 있는지 나타낸다.

- `GET`: 자원을 가져온다
- `POST`: 데이터를 보낸다
- `DELETE`: 자원을 삭제한다
- `PUT/PATCH`: 자원을 수정한다

과제 설명에 `UPDATE`라는 표현이 있더라도 실제 HTTP 표준 메서드는 `PUT` 또는 `PATCH`다.

이번 tiny 예제는 교육용이라 `GET` 중심으로 설명된다. 이유는 HTTP의 기본 구조를 보여 주기에 가장 단순하기 때문이다.

왜 중요한가:
- method는 요청 라인의 첫 번째 토큰이다.
- tiny는 보통 `GET` 중심으로 다루지만, proxy는 method를 읽고 전달하는 감각이 필요하다.

tiny/proxy에서 어디 쓰이는가:
- tiny의 method 검사
- proxy의 요청 중계

자기점검 질문: tiny가 모든 method를 다 지원하지 않아도 학습용으로 충분한 이유는 무엇인가?

## 프록시 서버
프록시 서버는 클라이언트와 원서버 사이에 위치해 요청과 응답을 대신 주고받는다. 중요한 점은 proxy가 단순히 "복사기"가 아니라는 것이다. 요청을 읽고, 목적 서버를 파악하고, 새 연결을 만들고, 응답을 다시 전달해야 한다.

책 관점에서 proxy는 클라이언트 코드와 서버 코드가 한 프로그램 안에서 만나는 지점이다. 따라서 소켓 흐름을 양쪽 관점에서 모두 이해해야 한다.

왜 중요한가:
- Proxy Lab의 직접적인 대상이다.
- tiny보다 설계 난도가 올라가는 이유를 보여 준다.

tiny/proxy에서 어디 쓰이는가:
- 브라우저 쪽에서는 서버
- 원서버 쪽에서는 클라이언트
- 캐시가 붙으면 저장소 관리자 역할까지 추가

자기점검 질문: proxy는 tiny보다 어떤 책임이 늘어나며, 왜 그 때문에 구조가 더 복잡해질까?

## REST API
REST API는 HTTP를 사용해 자원을 다루는 설계 방식이다. 이번 과제가 REST 서버를 만드는 것은 아니지만, URI와 method를 "자원과 동작" 관점에서 해석하는 훈련에 도움이 된다.

왜 중요한가: HTTP를 단순 문자열이 아니라 의미 있는 요청으로 보는 감각을 길러 준다.

tiny/proxy에서 어디 쓰이는가:
- tiny는 완전한 REST API 서버는 아니지만 요청 해석의 출발점이 된다.
- proxy는 API 요청도 결국 HTTP 요청으로 취급해 중계할 수 있다.

자기점검 질문: tiny는 웹서버 예제이지만 왜 REST API 서버 전체를 대표한다고 보기는 어려울까?

## OSI 7계층
이번 과제에서 OSI 7계층을 세부적으로 외울 필요는 없다. 하지만 적어도 "내가 지금 보고 있는 문제가 어느 층의 문제인지" 구분하는 감각은 필요하다.

- 애플리케이션 계층: HTTP
- 전송 계층: TCP/UDP
- 네트워크 계층: IP

이 정도만 잡아도 충분히 실용적이다.

왜 중요한가:
- 브라우저 에러가 났을 때 문제 위치를 더 잘 좁힐 수 있다.
- HTTP 파싱 문제와 TCP 연결 문제를 섞지 않게 해 준다.

tiny/proxy에서 어디 쓰이는가:
- 연결 자체 실패는 TCP/IP 쪽 문제일 수 있다.
- 연결은 됐지만 응답 형식이 이상하면 HTTP 쪽 문제일 수 있다.

자기점검 질문: 요청이 실패했을 때 "연결 실패"와 "HTTP 응답 실패"를 어떻게 구분할 수 있을까?

## CDN
CDN은 여러 위치에 콘텐츠를 복제해 두고 사용자 가까운 곳에서 빠르게 제공하는 분산 캐시 시스템이다. Proxy Lab의 캐시는 훨씬 작은 규모이지만, 아이디어는 닮아 있다.

왜 중요한가: 캐시가 단지 "빨라지는 기능"이 아니라 네트워크 비용과 원서버 부하를 줄이는 구조라는 점을 이해하게 해 준다.

tiny/proxy에서 어디 쓰이는가:
- proxy의 객체 캐시는 작은 로컬 CDN처럼 생각할 수 있다.
- 원서버가 내려가도 캐시가 있으면 일부 응답이 가능해지는 이유를 설명할 수 있다.
