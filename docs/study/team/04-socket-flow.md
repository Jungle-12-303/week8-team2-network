# 04. 소켓 흐름 이해하기

## 소켓 흐름을 먼저 잡아야 하는 이유
많은 초심자가 함수 이름은 외우지만 흐름을 잃는다. 실제로 중요한 것은 함수 하나하나보다 "서버는 언제 기다리고, 언제 읽고, 언제 응답하는가"라는 전체 리듬이다.

소켓 흐름을 이해한다는 것은 다음을 머릿속에 그릴 수 있다는 뜻이다.

- 서버는 요청을 기다리는 입장이다
- 클라이언트는 연결을 시작하는 입장이다
- 연결이 된 뒤에는 양쪽 모두 읽고 쓸 수 있다
- 연결이 끝나면 소켓을 닫는다

## 서버의 기본 흐름
서버는 보통 아래 순서로 움직인다.

1. `socket`
2. `bind`
3. `listen`
4. `accept`
5. `read/write`
6. `close`

이 순서를 보면 서버는 두 단계를 갖는다.

- 준비 단계: `socket`, `bind`, `listen`
- 요청 처리 단계: `accept`, `read/write`, `close`

준비 단계는 서버가 살아 있는 동안 한 번 또는 드물게 수행된다. 요청 처리 단계는 클라이언트 연결마다 반복된다.

## 클라이언트의 기본 흐름
클라이언트는 보통 아래 순서로 움직인다.

1. 이름 또는 주소 준비
2. `socket`
3. `connect`
4. `write`
5. `read`
6. `close`

여기서 서버와 가장 큰 차이는 `bind/listen/accept` 대신 `connect`가 나온다는 점이다.

## 서버 흐름을 더 자세히 보기

### 1. `socket`
운영체제에게 "통신용 소켓을 하나 열어 달라"고 요청하는 단계다. 이 시점에는 아직 어느 포트에서 기다릴지, 누구와 통신할지는 정해지지 않았다.

이 단계에서 흔한 오해:
- `socket`만 하면 바로 통신이 시작된다고 생각하기 쉽다.
- 실제로는 아직 준비만 된 상태다.

### 2. `bind`
이 소켓을 어떤 주소와 포트에 묶을지 정한다. 서버는 자신이 요청을 받을 포트를 운영체제에 등록해야 한다.

이 단계에서 흔한 오해:
- 서버가 자동으로 포트를 "알아서" 고른다고 생각하기 쉽다.
- 하지만 과제에서는 보통 인자로 받은 포트에 명시적으로 바인딩한다.

### 3. `listen`
이제 이 소켓이 들어오는 연결을 기다리는 서버 소켓이라는 사실을 운영체제에 알린다.

이 단계에서 중요한 감각:
- `listenfd`는 실제 데이터를 주고받는 소켓이 아니라 "연결을 받아들이는 입구"다.

### 4. `accept`
대기 중인 연결 하나를 꺼내서 실제 통신용 소켓을 만든다. 여기서 새로 얻는 소켓이 `connfd`다.

이 단계가 중요한 이유:
- 서버는 계속 같은 `listenfd`로 새 연결을 받는다.
- 동시에 각 클라이언트와의 실제 대화는 다른 소켓에서 이뤄진다.

### 5. `read/write`
이제부터 실제 요청/응답이 오간다. tiny에서는 HTTP 요청을 읽고 응답을 만든다. echo server에서는 읽은 바이트를 다시 쓴다.

### 6. `close`
해당 연결을 정리한다. iterative server에서는 요청 하나를 처리한 뒤 연결을 닫고 다시 `accept`로 돌아간다.

## 클라이언트 흐름을 더 자세히 보기

### 1. 이름 또는 주소 준비
사용자는 URL을 입력하지만, 프로그램은 결국 호스트와 포트 정보를 알아야 한다.

### 2. `socket`
클라이언트도 서버와 마찬가지로 소켓을 만든다.

### 3. `connect`
대상 서버와 연결을 시도한다. 이 시점에 비로소 "누구와 대화할지"가 정해진다.

### 4. `write`
요청을 보낸다. HTTP라면 request line, headers, 필요시 body를 보낸다.

### 5. `read`
응답을 읽는다.

### 6. `close`
대화가 끝나면 연결을 닫는다.

## 왜 `listenfd`와 `connfd`가 분리되는가
이 부분은 초심자가 가장 헷갈리는 지점이다. 서버는 문 하나만 들고 있으면 될 것 같지만, 실제로는 "새 손님을 받는 문"과 "이미 들어온 손님과 대화하는 자리"가 분리되어야 한다.

`listenfd`는 새로운 연결 요청을 계속 받아야 한다. 이미 들어온 손님과 이야기하는 동안에도 다음 연결을 받을 준비를 유지해야 하기 때문이다. 그래서 `accept`가 별도의 연결 소켓을 반환한다.

## Tiny에서 보이는 소켓 흐름
tiny의 `main`은 소켓 흐름을 가장 단순하게 보여 준다.

- `Open_listenfd(argv[1])`: 리스닝 소켓 생성
- 무한 루프 안에서 `Accept(...)`: 연결 수락
- `doit(connfd)`: 요청 처리
- `Close(connfd)`: 연결 종료

여기서 중요한 점은 `listenfd`는 루프 바깥에 있고, `connfd`는 루프 안에서 요청마다 새로 생긴다는 점이다.

## Proxy에서 보이는 소켓 흐름
proxy는 tiny보다 복잡하다. 왜냐하면 한 프로그램 안에 아래 두 흐름이 모두 있기 때문이다.

- 브라우저 쪽을 향한 서버 흐름
- 원서버 쪽을 향한 클라이언트 흐름

즉 proxy는 `accept`한 뒤 일이 끝나는 것이 아니라, 그 다음 다시 원서버에 `connect`해야 한다. 이것이 tiny와 proxy의 구조 차이를 만든다.

## TCP 스트림 위에서 읽는다는 뜻
소켓 흐름을 이해할 때 같이 잡아야 하는 것이 "한 번 읽으면 하나의 요청이 완성된다"는 착각을 버리는 것이다. TCP는 바이트 스트림이므로, 코드에서는 보통 request line을 먼저 읽고, 그 다음 헤더를 빈 줄이 나올 때까지 읽는다.

이 감각이 없으면 HTTP 요청 파싱이 어렵게 느껴진다.

## 흔한 실패 패턴
- `listenfd`와 `connfd`를 같은 변수처럼 이해함
- `accept`가 데이터를 읽는 함수라고 착각함
- `read` 한 번이면 요청 전체가 다 온다고 기대함
- 클라이언트 흐름과 서버 흐름을 뒤섞음
- proxy에서 `accept` 이후 원서버 연결 단계를 빼먹고 생각함
