# 09. Proxy 구현 로드맵

이 문서는 구현 정답을 제공하지 않는다. 대신 Proxy Lab을 어떤 순서로 이해해야 부담이 줄어드는지 설명한다. Proxy Lab은 기능이 많아 보여서 막막하지만, 실제로는 `순차 처리 -> 병렬 처리 -> 캐시`로 쪼개 보면 구조가 보인다.

## 왜 이 순서가 중요한가
프록시 과제를 처음 보면 병렬 처리와 캐시가 먼저 눈에 들어온다. 하지만 둘 다 "기본 전달이 된다"는 전제 위에 올라가는 기능이다. 따라서 첫 번째 목표는 멋진 구조가 아니라 "요청 하나를 제대로 전달하는 것"이어야 한다.

이 순서를 바꾸면 생기는 문제:
- 기본 전달도 안 되는데 스레드 문제와 씨름하게 된다
- 캐시가 틀린 것인지 기본 요청 전달이 틀린 것인지 구분이 안 된다
- 디버깅 포인트가 너무 많아진다

## 1단계: 순차 처리
이 단계에서는 프록시의 최소 책임을 이해한다.

핵심 질문:
- 브라우저 요청을 어디서 읽는가
- 원서버 주소는 어떻게 알아내는가
- 원서버에는 어떤 소켓으로 연결하는가
- 응답은 어떻게 다시 브라우저에 보내는가

이 단계에서 설명할 수 있어야 하는 것:
- 프록시는 브라우저 앞에서는 서버다
- 프록시는 원서버 앞에서는 클라이언트다
- 요청 하나가 끝날 때까지 양쪽 연결을 모두 관리해야 한다

`Basic` 테스트는 바로 이 단계가 맞는지 확인한다. 텍스트 파일뿐 아니라 바이너리 파일도 제대로 전달되어야 하므로, "문자열처럼만 다뤄도 된다"는 오해를 버려야 한다.

## 순차 처리 단계에서 자주 놓치는 포인트
- 요청을 읽고 끝나는 것이 아니라, 읽은 요청을 원서버로 다시 보내야 한다
- 응답을 일부만 읽고 끝내면 안 된다
- 연결 방향이 하나 더 늘어났다는 점을 계속 의식해야 한다

## 2단계: 병렬 처리
순차 처리 프록시는 한 요청이 오래 걸리면 다른 요청을 처리하지 못한다. 드라이버는 응답하지 않는 `nop-server` 요청을 하나 걸어 둔 상태에서도 tiny 요청을 계속 처리할 수 있는지 확인한다.

이 단계에서 학습해야 하는 것은 "스레드를 쓴다"라는 사실 자체보다, 왜 요청 처리 단위를 분리해야 하는가다.

핵심 질문:
- 어떤 지점에서 프록시 전체가 블록될 수 있는가
- 요청 하나를 독립 작업으로 분리한다는 것은 무슨 의미인가
- 스레드를 사용하면 어떤 자원이 공유되는가

이 단계에서 설명할 수 있어야 하는 것:
- iterative 구조가 왜 병목이 되는가
- concurrent 구조는 무엇을 개선하는가
- 연결 하나당 작업 단위를 나누는 사고방식

## 병렬 처리 단계에서 자주 놓치는 포인트
- 병렬 처리의 목표는 단순히 "빠르게"가 아니라 "서로 막지 않게"다
- 스레드가 늘어나면 로그, 캐시, 공유 버퍼 같은 자원 문제가 새로 생긴다
- 느린 서버 요청과 정상 요청을 분리해서 생각할 수 있어야 한다

## 3단계: 캐시
캐시는 기본 전달이 안정적으로 된 뒤에 붙이는 것이 맞다. 캐시는 기능을 하나 추가하는 것이 아니라, 프록시의 상태를 "무상태 중계자"에서 "상태를 저장하는 중계자"로 바꾼다.

핵심 질문:
- 어떤 응답을 캐시에 넣을 것인가
- 얼마나 큰 객체를 저장할 것인가
- 같은 요청이 다시 오면 어떻게 재사용할 것인가
- 여러 스레드가 캐시에 동시에 접근하면 어떻게 할 것인가

이 단계에서 설명할 수 있어야 하는 것:
- cache hit과 cache miss
- 객체 크기 제한의 의미
- 원서버가 꺼져도 캐시 응답이 가능한 이유
- 캐시가 동시성과 만나면 왜 동기화 문제가 생기는지

## 캐시 단계에서 자주 놓치는 포인트
- 캐시는 응답을 "다 받은 뒤"에야 의미 있게 저장할 수 있다
- 잘못된 캐시는 아예 없는 캐시보다 디버깅이 어렵다
- 캐시는 성능 기능이지만, 드라이버에서는 기능 정확성으로 먼저 검증된다

## driver와 로드맵의 대응 관계

### `Basic`
최소 프록시 전달이 맞는지 본다.

### `Concurrency`
요청 하나가 전체를 막지 않는지 본다.

### `Cache`
응답을 저장하고 재사용할 수 있는지 본다.

이 순서가 곧 구현 순서이자 학습 순서다.

## Tiny와 연결해서 보는 Proxy 로드맵
Tiny를 이해한 사람은 프록시를 볼 때 아래 비교를 계속 떠올리면 좋다.

- tiny의 `Accept` 이후에는 요청 처리만 하면 된다
- proxy의 `Accept` 이후에는 요청 처리 + 원서버 연결 + 중계가 필요하다
- tiny는 응답을 직접 만든다
- proxy는 응답을 읽고, 전달하고, 때로는 저장한다

즉 proxy는 tiny의 대체물이 아니라 tiny 위에 새 책임이 올라간 구조다.

3. 캐시는 왜 단순 부가 기능이 아니라 프록시의 상태 관리 문제를 여는 기능인가?
