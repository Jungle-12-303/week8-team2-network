# 01. 학습 로드맵

## 왜 이 순서로 공부하는가
CSAPP 11장은 "네트워크 프로그래밍을 위한 재료"와 "그 재료를 이용해 실제 서버를 만드는 과정"이 이어지는 장이다. 책을 읽다 보면 함수 이름과 코드가 먼저 눈에 들어와서 순서를 놓치기 쉽다. 이번 학습에서는 책의 흐름을 아래처럼 다시 정리해서 본다.

1. 네트워크 프로그래밍이 어떤 문제를 푸는지 이해한다.
2. 소켓과 주소, 포트, 연결 같은 기본 재료를 이해한다.
3. 가장 단순한 요청/응답 구조인 echo server를 떠올린다.
4. tiny를 통해 HTTP 서버의 최소 구조를 본다.
5. proxy를 통해 서버이면서 동시에 클라이언트인 프로그램을 본다.
6. 마지막으로 동시성과 캐시를 붙여 실제 과제 수준으로 확장한다.

이 순서를 따라가면 "함수 외우기"가 아니라 "왜 이 함수가 여기서 필요한가"를 기준으로 학습할 수 있다.

## 이번 학습의 최종 목표
이번 주 학습의 최종 목표는 아래 5가지를 한 줄로 연결해서 설명할 수 있는 상태가 되는 것이다.

1. 소켓 이해하기
2. 에코 서버 이해하기
3. CSAPP 코드 기반으로 `tiny/tiny.c`, `tiny/cgi-bin/adder.c` 완성하기
4. 11.6c, 11.7, 11.9, 11.10, 11.11을 스스로 공부하고 설명하기
5. `proxy.c`를 완성해 순차 처리, 병렬 처리, 캐시를 구현하기

핵심은 각 단계를 따로 외우는 것이 아니라, 앞 단계가 뒤 단계의 발판이라는 점을 이해하는 것이다.

## CSAPP 11장을 이번 과제와 연결해서 보기

### 11.1~11.4에서 잡아야 하는 것
이 구간은 네트워크 프로그래밍의 기초 재료를 설명하는 부분으로 보면 된다.

- 네트워크의 큰 구조
- 클라이언트-서버 모델
- 프로세스가 서로 통신하는 방식
- IP 주소와 포트의 의미
- 소켓 인터페이스
- DNS와 주소 해석

이 부분은 직접 서버를 만들기 전에 반드시 머릿속에 있어야 하는 전제다. 이 전제가 약하면 tiny 코드가 함수 호출 나열처럼 보이고, proxy는 더더욱 복잡하게 보인다.

### 11.5 이후에서 잡아야 하는 것
이 구간부터는 실제 코드를 통해 서버를 구성하는 방식이 나온다.

- 연결을 기다리는 서버
- 요청을 읽는 루틴
- 응답을 쓰는 루틴
- 웹서버의 정적/동적 응답
- 반복 서버(iterative server)와 동시 서버(concurrent server)

tiny는 이 구간을 압축해서 보여 주는 예제다. proxy는 tiny보다 더 복잡한 구조를 요구하는 실습 확장판이다.

## 이번 학습에서 우선순위를 높게 둘 것
이번 과제는 "네트워크 전체"를 공부하는 시간이 아니라 "네트워크 프로그래밍"을 공부하는 시간이다. 따라서 아래 개념은 깊게 잡아야 한다.

- BSD 소켓
- IP, 포트, 주소 해석
- TCP와 HTTP의 관계
- file descriptor
- 클라이언트 흐름과 서버 흐름의 차이
- 정적 응답과 동적 응답의 차이
- iterative server와 concurrent server의 차이
- proxy의 중계 역할
- 캐시의 필요성

반대로 아래 내용은 이름과 역할 정도만 이해하면 충분하다.

- OSI 7계층의 모든 세부 계층
- 라우팅과 패킷 포워딩의 물리적 디테일
- 실제 인터넷 장비 레벨의 동작

## 소켓 -> echo -> tiny -> proxy의 연결

### 소켓
소켓은 네트워크 통신의 입구다. 네트워크 프로그래밍의 거의 모든 동작은 결국 "소켓을 만들고, 연결하거나 기다리고, 읽고, 쓰고, 닫는" 흐름으로 귀결된다.

### echo server
echo server는 그 입구를 가장 단순하게 사용한 예제다. 요청을 읽고 그대로 다시 보내 준다. 아직 HTTP도 없고 파일도 없고 CGI도 없다. 하지만 요청/응답의 본질은 이미 다 들어 있다.

### tiny
tiny는 echo server 위에 HTTP 해석을 올린 서버다. 요청을 그냥 돌려주지 않고, 요청 라인과 URI를 읽고, 파일을 보내거나 CGI를 실행한다. 즉, "읽고 쓰는 것"에서 "읽고 해석하고 응답하는 것"으로 넘어간다.

### proxy
proxy는 tiny와 또 다르다. 브라우저에게는 서버처럼 보이지만, 원서버에게는 클라이언트처럼 행동해야 한다. 즉 한 프로그램 안에서 서버 흐름과 클라이언트 흐름이 동시에 등장한다. 여기에 병렬 처리와 캐시까지 들어간다.

## Tiny와 Proxy를 학습 마일스톤으로 보기

### Tiny를 끝냈다는 뜻
- HTTP 요청/응답의 최소 구조를 이해했다
- 정적 콘텐츠와 동적 콘텐츠의 차이를 이해했다
- 브라우저와 서버가 문자열 기반 프로토콜로 대화한다는 점을 이해했다
- 서버 코드가 시스템 호출, 파일 I/O, 프로세스 실행을 묶어서 동작한다는 점을 이해했다

### Proxy를 끝냈다는 뜻
- 서버와 클라이언트 역할을 한 프로그램 안에서 함께 설계할 수 있다
- 느린 요청이 전체 서비스를 막는 문제를 이해한다
- 캐시가 왜 성능과 가용성을 바꾸는지 이해한다
- 네트워크 프로그램에서 동시성과 공유 상태를 함께 고민해야 함을 안다

## 권장 학습 운영 방식
이번 학습은 "개념 후 실습" 흐름으로 보는 것이 가장 효율적이다.

1. 먼저 개념 문서를 읽고 용어를 맞춘다.
2. 소켓 흐름과 echo/tiny 차이를 이해한다.
3. tiny 코드를 직접 따라가며 요청 하나가 어떻게 처리되는지 본다.
4. 연습문제는 정답 찾기보다 개념 확인용으로 사용한다.
5. proxy 문서로 넘어가 tiny에 없던 책임이 무엇인지 정리한다.
6. 마지막으로 실습 체크리스트와 자기점검 질문으로 빈칸을 확인한다.
